## 0. 一句话定位

为「不规模化、高粘性」的社区家庭做饭服务者（首位用户：全职妈妈桃子）做一个**老板侧的"经营大脑"**：
用**语音和粘贴**把散落在微信里的订单收拢成一张清单，用 **AI 从她自己做过的菜里**生成不重复的每日菜单，并把打包、配送状态串成一条防漏防错的流水线——**顾客侧零改动**，让她每天少花脑子、少漏单、少头疼。

---

## 0.5 愿景

> 让街坊邻里的**小市民经济**重新活起来：每个人拿出自己的一技之长，为邻居生产一份吃食、一项服务；与此同时，也消费别人的手艺。**人人各尽所能、互为供需**，把一个个家庭重新织回彼此的日常——长出一个**有烟火气、有人情味的社区**，而不是资本主义大生产体系里那种被标准化、被疏远的冷淡交易。我们希望它最终像古文写的市井那样热闹、喧腾如沸。

**项目口号：廛闬扑地，歌吹沸天。**（街市屋舍铺满大地、歌声乐声喧腾冲天；"廛闬"太生僻，故**只作口号、不入产品名**。）

落到产品判断：

- 我们服务的不是冷冰冰的交易，而是**熟人之间的信任与连接**；工具帮这些"生产者"被看见、把街坊织成网。
- 因此**首位用户从桃子起步，但产品天生面向多类社区生产者**（呼应 §2 可复制人群、多商家 schema）。
- 也因此**命名要带 neighborhood 的意味、中文要接地气**（已定：街坊味 / kith）。

**长期愿景（远期，与 V1 无关，不影响当前范围）**

更长远的设想：不止做饭——**每位居民都能写下自己能提供什么**（做饭 / 家教 / 理发 / 维修……）；当邻里需要某种服务时，能**定位到身边能提供的人**；我们再**按每个提供者的实际需要，给他配一套趁手的工具**（桃子这套做饭"经营大脑"，就是第一个实例）。

- 一句话：**邻里能力目录 + 按需匹配 + 因人定制工具**。
- 它也解释了当前几个架构取向（虽 V1 不做）：多租户从第一天留好；每个提供者的"自我描述"天然是一份松散、个性化的画像（与 §6.0「类 skill 画像」同源）。
- **边界**：这是北极星式远景，**不在 V1 范围**；当前一切设计仍以桃子的做饭场景为准。

## 1. 背景与问题

### 1.1 首位用户：桃子

- 全职妈妈，带两娃；奶奶（约 70 岁）帮打包、配送、晒衣；老公在附近工厂上班。
- 2024-03-08 起步：幼儿园门口早餐摊 → 微信群早餐 → 因太累砍掉早餐，转**周一至周五中晚餐**，至今约 2 年多。
- 微信群「桃子的健康早午餐」200+ 人；日均约 **20 份**（峰值 30），统一 **30 元/份**（4 菜 1 汤，不分大小份/有无汤）。
- **护城河 = 食材品质 + 熟人信任**：牛肉 60–70 元/斤，从不在普通菜市场买；停业期间顾客非她不订、天天催开工。曾因产能拒掉公司「30 份/餐」的大单——**她明确不想规模化、不想升级成大锅**。
- 利润薄、保本心态：自家四口也吃这锅饭，单餐约 8 份即覆盖成本，毛利率粗估 ~40%，**她明确不在意精算成本**（"穷不了富不到哪去"）。

### 1.2 一天的劳动链路（现状）

| 时间 | 环节 | 现状与摩擦 |
|---|---|---|
| 前一晚 | 发次日菜单 | 群里发"4 菜 1 汤"。**她最头疼的一步**：怕和昨天重复、肉就那几样、靠记忆即兴 |
| 前一晚/清晨 | 收单 | 主走**群接龙**；部分顾客**私聊**下单，固定客她手动补数字防漏 |
| ~8:00 | 买菜 | 凭感觉、偏多买；部分打折菜前一晚买 |
| 10:00 | 备菜 | 洗切，看着菜单定量 |
| 10:50–11:30 | 炒制 | |
| 11:30 起 | 打包 | 本人 + 奶奶 |
| 12:00–12:20 | 午餐配送 | 按楼栋绕行；本小区送到门口，外小区放大门口架子；**电梯常等 10 分钟+** 是主要不确定性 |
| 傍晚 | 晚餐配送 | 送达时间因人而异（小学 6:30 / 初中 7:30 才到家），**部分需二次加热再送**；她自己约 8 点才吃饭 |

### 1.3 痛点分级（沿用访谈提纲的 P0 / P1 / 伪痛点框架）

**P0 — 强痛点（高频、有真实代价、她愿意试，MVP 主攻）**

1. **私聊下单遗漏**：接龙之外的私聊单全靠脑记，经常漏。访谈当天就因忘记拖到晚上 8 点才送；暴雨那天奶奶把餐错送到 26B 引发顾客投诉、对方甚至要求定赔偿规则。
2. **每日菜单生成**：她亲口"最头疼"，怕跟昨天重、"肉就那几样"。关键认知——"**我发过的都是我做过的**"，她说过的菜天然就是可做菜库，去重的真实约束在主料层（设计见 §6.2）。

**P1 — 中痛点（真烦但已习惯 / 上量才明显 / 需长期数据）**

3. **收款漏收**：红包 24 小时过期且无提醒（转账有 2 小时提醒），丢过 4 月份的红包。
4. **固定周期预订**：每周固定客不该天天接龙；现成「群接龙精选」不照顾"一周/一月一订"，正是缝隙。
5. **送餐待送清单**：100% 靠脑记，偶漏；漏送/错送的根因就是没清单。
6. **菜量/价格差异化**：被产能锁死才统一价；杂事松绑后才有余力做差异化。

**伪痛点 / 不该软件化（明确不做，详见 §4.2）**

- 成本利润核算（她不在意 + 会制造焦虑）
- 家务/家庭分工可见化（敏感、她很佛系）
- 配送路线优化（她对每个小区了如指掌、不用地图）
- 忌口记忆做成卖点（现在几乎无忌口）→ 降级为顾客备注字段

### 1.4 付费与工具习惯（硬约束）

- **强烈抗拒月费**，明确偏好**按单抽成**（类比信用卡利息："有需求就该付费、哪有免费的"）。
- 活在微信里；**偏好语音**；讨厌打字、小字、多步骤、注册、广告。
- 对订单数据**无隐私顾虑**（地址房号群里已公开）。
- 愿当首批试用、群内推广、引荐其他宝妈（糖水/蛋糕/寿司/饺子/面条，同模式），并开放厨房供实地观察。

## 3. 产品原则

1. **顾客侧零迁移**：MVP 不要求 200 个顾客改变任何习惯，订单仍走原微信群。
2. **语音优先、最少打字**：核心录入靠语音/粘贴；大字、少步骤、能不注册就不注册。
3. **只用她做过的菜**：菜单生成是"对她自己菜库的不重复重组"，**绝不发明她不会做的菜**。
4. **不制造焦虑**：默认不展示成本/利润、不做家庭分工，除非用户主动开启。
5. **熟人经济友好**：工具帮她"被看见"、体面沟通，而非把邻里关系变成冷冰冰的平台交易。
6. **多商家 schema、单商家 UX**：底层可复制，体验先打磨到桃子一个人极致顺手。
7. **先把价值做出来，盈利搁置**：免费试用换留存；是否 / 如何收费留到最后单独讨论（见 §5）。

## 4. 范围

### 4.1 做什么（按里程碑，详见 §9）

- **MVP（M1）**：记单（接龙粘贴导入 + 语音补私聊 + AI 结构化）、AI 菜单生成、按楼栋送餐清单（勾选 + 收款状态）。
  - **MVP 边界（重要）**：全程**只有桃子一个使用者**，**不含任何顾客侧界面、不做顾客点单、不接任何支付**。顾客仍走原微信群，桃子把信息录进来；收款仅由她手动标状态。顾客侧一切（含只读看菜单）均属 M3/V1。
- **体验闭环（M2）**：厨房免手语音、收款催收提醒、固定客一键复制、顾客沟通文案模板。
- **双边与订阅（M3 / V1 核心）**：顾客端点单 + 周/月订阅（**下单即付 / 盈利搁置**，见 §5）。
- **复制与抽成（M4）**：多商家自助开通、按单抽成计费、引荐其他宝妈。

### 4.2 明确不做（本期）

| 不做项 | 原因 |
|---|---|
| 成本/利润精算与报表 | 她明确不在意，且提纲警告会制造焦虑；留作 V1+ 面向"事业型"卖家的可选模块。注：**"买多少"的采购量估算属采购辅助、与成本无关，是另一回事，见 §6.9 候选** |
| 家务/家庭分工与时间账本 | 家庭关系敏感，她很佛系，不希望被软件化 |
| 配送路线/地图优化 | 她对每个小区了如指掌，纯属伪需求 |
| 顾客忌口/偏好的重型记忆系统 | 当前几乎无忌口；降级为顾客备注字段即可 |
| 规模化/中央厨房/大锅产能 | 与她"不想升级"的意愿冲突，也偏离目标人群 |
| 强制注册/账号体系（顾客侧） | 抬高迁移成本；V1 顾客端尽量走微信一键授权 |

## 5. 商业模式（❓搁置 — 留到最后单独讨论）

> **盈利模式整体打问号、暂不决策。** MVP **完全不涉及**（不收费、不接支付、不写任何计费/抽成）；**V1 也暂不纳入**——"点单即付 / 按单抽成"等都搁置，放到产品跑通、有真实留存后再单独议。本节不作为当前设计依据。

- **MVP / 试用期**：完全免费，零支付、零计费。此阶段大概率还没完成小程序备案，故通过 **H5 / 微信体验版（白名单）等"非正式发布"渠道**给桃子试用（见 §10）。目标只有一个——换取"她每天真用"的留存信号。
- **V1**：以工具价值为先，**不预设收费**；是否、以及如何盈利留到最后讨论。
- **仅作记录、非决策**：她抗拒月费、对"1 毛/单"或"1%"抽成可接受（类比信用卡利息）；收款路径分析见 §10.2。这些都先存档，不在此拍板。
- 增长：靠她在群内推广 + 引荐其他宝妈，零获客成本的熟人扩散。

## 6. 功能需求

> 标注：**[MVP]** = M1 必交付；**[M2]/[V1]/[M4]** = 后续里程碑。每条尽量配「用户故事 + 验收要点」。

### 6.0 经营画像 · 配置初始化 **[留白 · 待专门讨论]**

> **本节先留白**——设计尚未敲定，需要专门讨论后再补。这里只记目标与核心难题，不预设方案。

- **目标**：商家首次开通时，用 **AI 引导式问答**确立"怎么出餐 / 怎么运营"的配置（几菜几汤、按天还是按周、服务日……），并沉淀成可被产品使用的画像。
- **核心难题**：现在只摸了桃子一个样本，而生产者高度异质（做饭有"荤素结构"，糖水 / 理发 / 家教等可能根本没有）——配置本质是**松散、个性化**的信息。怎样**既能结构化到驱动确定性逻辑（菜单生成、排期），又不把不同生产者塞进同一套死板字段**，是待解的关键。
- **倾向（未定）**：用"类 AI skill"的思路——薄薄的通用结构外壳 + 自由文本内核；结构层随样本增多再演进。**具体形态留到专门讨论**。
- 关联：这也是 §0.5「长期愿景」里"每位提供者写自我描述"的近亲，远期会更重要。

### 6.1 记单 · 订单收拢 **[MVP]**

把散落的接龙与私聊单收拢成一张当日订单表。

录入有**两个并行入口**，对应她"既能直接贴接龙、又能随口说一句补"的真实习惯：

- **接龙粘贴导入（主入口）**：她把微信群接龙的文本整段粘贴进来，AI 解析成结构化订单（顾客、楼栋房号、份数、餐次、特殊要求），并与已有顾客做模糊匹配（"张阿姨""26B"自动归位），不认识的就新建。
- **文本/口述补充录入**：接龙之外的一切——私聊单、临时加单、改单——她在一个文本框里输入即可追加或修改。**输入既能打字、也能直接按手机输入法的语音键口述**（豆包/讯飞/系统输入法的语音输入对用户免费、甚至离线可用），因此 **MVP 不自建 ASR**；DeepSeek 仍负责把这段文字结构化成订单：
  > "26 楼 B 李叔加一份，少辣不要葱，还没付款"
  这恰好命中她最容易漏的环节（散落的私聊单），也契合她厨房没空打字、偏好语音的习惯。
- **可选增强（非 MVP）**：若日后想要"应用内一键长按说话"的体验，再接微信「同声传译」插件做应用内语音识别（免费，有配额：约 250 条/分钟、3 万条/天/小程序）。
- **手动微调**：任何一条都能一键改份数、改要求、删除。
- **防漏校验**：与"固定客名单"对照，提示"周三常订的王阿姨今天还没下单，要补吗？"

**用户故事**：作为桃子，我希望把接龙一粘、私聊一说，就得到一张完整不漏的当日订单表，这样我不用再靠脑子记谁点了什么。

**验收要点**：
- 粘贴一段真实接龙文本，解析准确率达到"她无需大改即可用"的程度（试用期人工校验）。
- 一句话语音可在 ≤3 秒内追加一条正确订单（楼栋、份数、要求三要素到位）。
- 全程不需要打字也能完成一次完整记单（允许个别微调）。

### 6.2 菜单生成 **[MVP]**

> 设计立场：这不是"AI 发明菜单"，而是**替她承担"在自己会做的菜里不重样"的排列负担**。因此采用**确定性选菜内核 + LLM 仅做润色**的分层设计——内核可被确定性单测覆盖（契合仓库 100% 覆盖率门禁），LLM 不参与"选哪道菜"的决策，从根上杜绝"推荐了她不会做的菜"。

**菜品池（dishes）**
- 每道菜的维度：分类（荤/素/汤/主食）、**主料**（牛肉/鸡/鱼/排骨/猪肉/豆制品/青菜…）、标签（清淡/费工…）、上次使用日期、使用频次。
- **主料维度是关键**：她的真实约束是"肉就那几样"，去重必须在主料层做，而不仅是菜名层。
- 冷启动见 §9 M0：一次性语音/快捷录入她的常用菜 + 日常增量积累，**不硬依赖群历史导出**（导出成功则作为加速项）。

**确定性选菜内核（不经过 LLM）**
- 组成「4 菜 1 汤」（荤素结构可配，默认 2 荤 2 素 1 汤）。
- 约束：
  - **只选池内菜，绝不发明新菜**；
  - **最近 N 天主料不重复**（默认按她"不想跟昨天一样"，N 可调）；
  - 单道菜最近 N′ 天不重复；
  - 费工菜（如煲汤）每日不超阈值，避免一天全是麻烦菜（呼应她"煲汤很麻烦"）；
  - 频次加权：常做的优先、冷门的偶尔翻新。
- **一周一次性排好**：一次生成周一至周五（含午/晚）的菜单，跨天满足上述去重，她每天直接执行（她明确想要）。
- **一键换菜**：对任意一道不满意，点一下即在满足约束的候选里替换，不用她自己想替代品。

**LLM 润色层（DeepSeek，可选、不决定选菜）**
- 菜名口语化润色；
- 节令/天气轻提示（"天热，今天的汤换清爽点？"）；
- 池子太小无法满足约束时，礼貌提示"可做的菜不够了，补几道？"；
- **一键发布文案**：把选定菜单写成"她本人语气"的群通知（菜名、价格、截止时间）。

**用户故事**：作为桃子，我希望晚上点一下就有明天（或整周）的菜单，都是我会做、又不和最近重样（尤其不老是同一种肉）的菜，这样我不用再每天对着冰箱发愁。

**验收要点**：
- 生成的菜单 100% 来自她的菜品池；
- 连续生成 7 天，主料在 N 天窗口内不重复、单菜在 N′ 天窗口内不重复；
- LLM 故障/超时时，确定性内核仍能产出可用菜单（LLM 仅降级润色）；
- 她"直接采用或小改即用"的比例是核心留存指标。

### 6.3 防漏送 / 防错送 **[MVP]**

> **真实约束（v0.4 修正）**：① 送餐时手里拿满了，**没法一单一勾**；她的自然颗粒度是**一栋记一次**。② **奶奶不用手机**，实际是桃子把某栋/某片交给奶奶送，桃子心里记"这栋送过了"。所以防漏防错不能靠"逐单打勾"，要顺着"按栋、手忙、奶奶离线"的现实。

把问题拆成"源头防错"和"收尾防漏"两段，逐单交互降到最低：

- **① 打包分拣视图（源头防错，最关键）**：出餐后给一张"这一趟要送什么"的**按楼栋汇总**（如 3A×2、3B×1、26B×1），照着这张分拣装篮——**在打包环节就把错误挡住**。暴雨错送 26B 那次本质是打包/交接装错，分拣视图比送达时再补救更治本。
- **② 灵活勾销·语音为主（粒度由她定）**：完成标记**不强制逐单、也不锁死整栋**——她说了算：可以说"**3A 送完了**"整栋勾掉，也可以说"**26B 送了**"勾单个门牌。手占着就**语音报**，腾出手也能点。
- **③ 按人分片 + 离线协作**：支持把某栋/某片**指派给奶奶**，桃子侧把"卓红这片交给奶奶"标成一个整体（已交接）；奶奶不需要装 App，桃子心里 + App 这一条记录兜住。
- **④ 收尾对账（缺口式提醒，不催时辰）**：一趟结束，App 按缺口提示"这趟 6 栋，26B 还没标送到哦"——**提醒的是遗漏，不是钟点**（呼应 §3 设计原则）。
- **送达时段 / 二次加热**：每单可带"午 / 晚 / 6:30 / 7:30 / 需加热"标记，晚餐分批不乱（时段来自她或顾客约定，不由 App 强加）。

> **关于贴标签**：给饭盒贴"楼栋房号"标签确实是防错送的好办法，但那是**线下动作、与软件无关**——本产品不做标签功能；软件最多在需要时导出一份可抄写的分拣清单，仅此而已。

**用户故事**：作为桃子，我希望出餐后有一张按楼栋分好的分拣单，装篮不装错；送的时候想报一栋就报一栋、想报一个门牌就报一个门牌（动动嘴），交给奶奶的那片标一下"已交接"，收工时它提醒我哪里还没送到——不用我一单一单点。

**验收要点**：
- 打包分拣视图按楼栋正确汇总。
- 完成标记同时支持"整栋"和"单门牌"两种粒度，且可语音，不强制逐单。
- 可把某片标为"交给奶奶"。
- 收尾能按缺口提示未送达处，零遗漏可核对。

### 6.4 顾客与备注（轻量）**[MVP]**

> **「轻量」= 只存一张够用的小卡片，不做重型 CRM**：不建顾客偏好/忌口推荐引擎（§4.2 已排除），不采集真实姓名、电话。这些字段绝大多数在 §6.1 记单时**自动沉淀**（她说一句"26B 李叔加一份"，系统就记下了 26B 这位顾客），她几乎不用专门去"管理顾客"。

- 顾客 = 称呼（张阿姨/李叔）+ 楼栋房号（如 26B）+ 分组（固定/散客）+ 默认份数 + 备注（口味/忌口/自带饭盒）+ 默认送达时段。
- **自家也是一位"顾客"**：桃子自家吃的也走同一条接龙，她自己留言如"桃子-5 份晚餐"。系统把"自家/本人"建成一个特殊顾客，记单时照常解析——这样份数、采购量都把自家算进去（呼应她"自家四口也吃这锅饭"）。
- **隐私原则**：以"称呼 + 楼栋房号"为主键，不强制采集真实姓名、电话（与访谈提纲的脱敏要求一致）。

### 6.5 收款闭环

- **[MVP]** 仅做**收款状态标记**：把每单标"未付 / 已付 / 已确认"，"未付"单可一眼汇总。**纯手动、不生成任何文案/提醒**。
- **[M2]** **催收提醒 + 文案**：针对红包易过期给"X 单待收、将过期"提醒，并一键生成体面的催收文案（与 §4.1、§9 的 M2 一致）。
- **[搁置 / 待定]** **下单即付**：顾客端点单即收款本可从源头根除漏单与漏收，但**涉及支付/盈利，整体搁置到最后讨论**（见 §5、§10.2）。即便日后做，也倾向"收款搭车不自建"。

### 6.6 顾客沟通文案模板 **[M2]**

为高情绪劳动场景预置"她本人语气"的话术：明日菜单通知、临时拒单/产能已满、延迟致歉、催收、价格不议价的体面回绝。她省得现场想词。

### 6.7 顾客端点单 + 订阅 **[V1 / M3]**

- 顾客在小程序里看菜单、下单；支持**周/月固定订阅**（每周一至五、午/晚），可暂停/恢复。（**"即付/支付"搁置**，见 §5）
- 老板侧自动汇总，免去手动补固定客数字。
- 仍保留"原微信群 + 接龙粘贴"作为并行通道，迁移渐进、不强迫。
- **若日后做收款，不自建**：走点对点、或搭车现成平台 / 微信电商收付通（见 §10.2），我们不做收款平台。

### 6.8 多商家复制 **[M4]**（抽成搁置）

- 其他宝妈可自助开通自己的"灶台"（独立菜品池、顾客、群）。
- 按单抽成计费与对账（**搁置**，随 §5 盈利决策一并定）。
- 引荐机制（桃子推荐有礼）。

### 6.9 采购清单 + 备菜量估算 **[P1 / M2 候选，待现场验证]**

> 严格定位为**采购 / 备菜辅助，与成本 / 利润完全无关**（§4.2 已排除成本核算，避免制造焦虑）：只答"买什么、买多少"，绝不显示"赚多少"。

分两层，价值与门槛不同：

- **采购清单（买什么）— 高价值、低门槛**：把次日菜单（4 菜 1 汤）的食材自动聚合成一张"要买的菜"清单，她照着买、不漏项。几乎不依赖精确标注。
- **用量估算（买多少）— 价值真、门槛在标注**：若她给菜品标了"每份大致用量"（如牛肉 100g、青菜 200g），则 用量 = 每份量 × 次日份数，汇总成采购量。
  - **门槛与红线**：标注与她"凭感觉买、买多、不用秤、嫌麻烦、每份量本就浮动"的习惯冲突——所以标注必须**一次性、可选、带默认值（按荤/素给缺省克重）、可随用随改 / 可学习**；没标也能退化成纯采购清单。
  - **输出用买菜的语言**（市斤 / 把 / 块，而非克），并考虑她"习惯多买 + 自家也吃"——给余量或让她设一个冗余系数。
  - **本质是估大概**：她 8 点买菜时订单还会加，估算只为"心里有数 + 留余量"，不追求精确。

**用户故事**：作为桃子，定好明天的菜单和份数后，我希望直接得到一张"要买什么、大概多少"的清单，照着买就行，不用每天在脑子里算。

**验证（关键，决定做不做"买多少"那半）**：在**现场观察那次**确认她是否愿意做一次性用量标注；愿意 → 做"买多少"，不愿意 → 只做"买什么"。

## 7. 信息架构与数据模型（多商家）

以 Payload 集合实现，全部以 `seller` 作租户隔离键。

| 集合 | 关键字段 | 备注 |
|---|---|---|
| `sellers` | 名称、服务小区/范围、群名、默认价格、状态；**＋经营画像字段（出餐结构 / 周期 / 服务日 / 配送方式 / 经营备注）待 §6.0 讨论后定** | 一个老板一条；MVP 仅桃子 |
| `operators`（auth） | 关联 seller、登录凭证 | 复用 Payload Auth；MVP 单用户 |
| `customers` | seller、称呼、楼栋房号、分组(固定/散客/**自家**)、默认份数、备注/忌口、默认送达时段、自带饭盒 | 不存真实姓名/电话；"自家"为特殊顾客(§6.4) |
| `dishes` | seller、名称、分类(荤/素/汤/主食)、**主料**、标签(清淡/费工)、上次使用日期、使用频次、食材 + 每份大致用量(均可选) | 一次性语音/快捷录入 + 增量；主料维度用于去重；每份用量供 §6.9 采购估算 |
| `menus` | seller、日期、餐次(午/晚)、dishes[]、发布文案、状态(草稿/已发) | 支持整周批量 |
| `orders` | seller、customer、日期、餐次、份数、特殊要求、价格、来源(接龙/私聊/手动)、送达时段、配送状态、收款状态、收款方式 | 核心表 |
| `subscriptions`【V1】 | customer、周期(周/月)、模板、暂停/恢复 | 固定周期预订 |
| `payments`【V1】 | order、金额、渠道(微信支付)、抽成 | 下单即付 + 抽成对账 |

- **送餐清单**在 MVP 阶段是 `orders` 的派生视图（按楼栋分组），不单建表。
- 历史菜单导入产出 `dishes` 与 `menus` 的初始数据（M0）。

- §6.3 送餐清单（按楼栋、勾选、送达时段、收款状态）。
- §6.4 轻量顾客与备注。
- 顾客侧零改动，免费试用。
- **交付/验收**：桃子连续一周用它完成"记单 → 出菜单 → 送餐清单"，漏单为 0；菜单"采用或小改即用"。

### M2 — 体验闭环
- 厨房免手语音（更稳的录音/转写、嘈杂兜底）。

- 固定客一键复制到当日订单。
- §6.6 沟通文案模板。

### M3 — 双边与订阅（V1 核心）
- §6.7 顾客端点单 + 周/月订阅（**下单即付 / 支付搁置**，见 §5）。
- 老板侧自动汇总，原群通道并行保留。
- **前置依赖**：微信支付商户开通、小程序主体/类目（见 §10）。

### M4 — 复制与抽成
- §6.8 多商家自助开通、按单抽成计费与对账、引荐机制。
- 引入第二、第三位宝妈（糖水/蛋糕/寿司）。

### 10.1 类目与资质策略（重要：决定 V1 钱要不要过平台）

> 已核对微信官方类目文档与开放社区帖（见文末 Sources）；**正式开类目前仍以微信最新官方文档 / 服务商确认为准**，规则常调整。

- **MVP / 不收款阶段**：小程序是"商家侧管理 / 门店管理展示工具"，注册在团队名下，走**工具类（办公工具 / 企业管理）类目**——官方明示**无特殊资质要求**，**不触发餐饮类目、不需要《食品经营许可证》**。这是合规最干净的落点。
  - 注：避开"商业服务"子类目，其部分子类按行业另需资质（如律师/会计等）。
- **一旦做成"顾客在小程序下单 + 钱经我们平台"（"小美团"模式）**：归入餐饮平台类目，门槛陡增——
  - **点餐平台**：平台主体需《增值电信业务经营许可证》(**EDI**)；**外卖平台**：另需《网络餐饮服务第三方平台备案》。平台类目本身不要求平台持食品证。
  - **两道真实阻塞**：① **EDI 本身**小团队很难申办（社区里大量开发者卡在这一步）；② 监管框架要求平台核验入驻商家的《食品经营许可证》，而**社区家庭厨房基本拿不到**。
- **由此的产品建议（待 V1 决策）**：即便到 V1，也优先让**收款是顾客↔商家点对点**（钱不经我们平台，我们只做下单 / 对账工具），从而**一直留在"工具类"、彻底避开 EDI 与商家食品资质**。代价是按单抽成无法自动从交易扣，需另想计费方式（工具订阅 / 打赏 / 线下结算）——与"她抗拒月费"的张力留到 V1 再权衡。

### 10.2 收款路径：别自建管道（看懂群接龙怎么收的款）

群接龙精选能"下单即付 + 自动抽成"，不是因为门槛低，而是因为它是一家有完整资质的平台公司在兜底：

- **主体与资质**：背后是广西尚食科技（2017 年起做接龙，有融资、年流水百亿级），自持平台级资质与**微信支付服务商**身份。
- **技术机制**：走微信支付「**电商 / 平台收付通**」——群接龙是**服务商**，每个团长/商家是挂在它下面的**二级商户（特约商户）**；顾客付款进二级商户，群接龙通过「**在线分账**」自动抽 **1%** 服务费（提现到银行卡，限额约 2 万/天）。
- **所以个体团长不用自己办微信支付商户号、不用办 EDI**——他们是"借伞"收款，1% 就是伞租。

**对我们的含义**：
1. 自己做收款平台 = 要么成为微信支付服务商（电商收付通），要么走点餐/外卖平台类目（EDI）——都是群接龙花多年 + 融资才搭起来的重资产，**不该是我们 MVP/V1 去抢的**（印证 §10.1）。
2. 聪明的做法：**收款"搭车"而非自建**——V1 若要下单即付，可让收款走群接龙这类现成平台，或日后接入微信「电商收付通」做二级商户分账；我们自己**始终留在工具层**。
3. **护城河重申**：群接龙精选已解决"下单即付 / 防漏单 / 防漏收"，但**不照顾"一周 / 一月固定订阅"，也没有老板侧大脑**（语音记私聊单、AI 菜单、按楼栋送餐清单）。我们的价值在它没做的这两块，**不在收款管道本身**。

## 11. 成功度量

- **北极星**：桃子**自发**使用频率（目标：工作日每天用，无需提醒）。
- 漏单数（工作日）→ 趋近 0。
- 每日"发菜单"耗时显著下降；菜单"采用/小改即用"比例 ≥ 目标。
- 红包/收款漏收次数下降。
- 她**主动**把工具推荐/引荐给 ≥1 位其他宝妈。
- 定性："被看见"的感受、对工具的依赖度（停用会不会难受）。

## 12. 非功能需求

- **可访问性/适老**：大字、少步骤、语音优先、避免多层级跳转与注册。
- **隐私**：顾客以"称呼 + 楼栋房号"为主键，不强采姓名电话；数据可导出/删除（呼应提纲"以后不用了数据怎么处理"）。
- **质量门禁**：遵循仓库 `pnpm verify`（lint + typecheck + **100% 覆盖率** + knip + build）；H5 端 Playwright e2e，小程序端手测——与 PLAN.md 一致。
- **可用性**：核心链路（记单/菜单/清单）在弱网与厨房单手操作下可用。
- **可复制性**：所有业务数据以 `seller` 租户隔离，新增商家零代码改动。
