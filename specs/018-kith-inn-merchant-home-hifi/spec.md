# Feature Specification: kith-inn-v1 商家今日工作台高保真重构

**Feature Branch**: `018-kith-inn-merchant-home-hifi`

**Created**: 2026-07-24

**Status**: Draft

**Input**: 依据指定工作树中的“今日工作台”高保真参考图、完整还原说明和 HTML 设计源，在不改变现有业务数据与能力边界的前提下重构商家首页。

## 项目作用域

本功能只重构 `kith-inn-v1` 商家端“今日工作台”的信息层级、视觉样式、状态文案和现有入口编排。页面仍使用既有商家身份、上海业务日期、午晚餐、菜单、订单汇总和导航能力，不改变后端接口、数据模型或认证方式。

范围不包含头像、伪造的微信系统胶囊、“我的”页面、聊天、AI、在线支付、新后端接口或静态演示数据。完整还原说明仅作为本地输入，不进入版本控制；高保真设计资产如需归档，应与实现拆成独立 PR。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 一眼完成今日经营判断 (Priority: P1)

作为商家桃子，我希望打开首页后按“待确认订单 → 午餐与晚餐 → 常用经营动作”的顺序看到真实经营信息，以便立即判断今天先做什么，并能直接进入对应操作。

**Why this priority**: 今日工作台是商家每天进入产品后的首要入口；信息层级、状态真实性和动作可达性必须作为同一个工作台体验闭环交付。

**Independent Test**: 使用有待确认订单、午餐预订中、晚餐待开放的数据进入首页，核对页面层级、真实文案与数量，并分别进入订单、手动加单、菜单、开放预订和配送清单；再覆盖空餐次、局部失败、整页失败和刷新状态，确认页面仍可理解和恢复。

**Acceptance Scenarios**:

1. **Given** 商家已登录且今日有待确认订单和两个餐次，**When** 打开今日工作台，**Then** 先看到品牌和问候，再看到仅在数量大于零时出现的待确认提醒、午晚餐卡片以及四个快捷入口。
2. **Given** 某餐次存在，**When** 商家查看餐次卡片，**Then** 菜单摘要、状态、已确认份数、价格和截止时间均来自当前真实数据，且“手动加单”始终可达，不受顾客登记截止或餐次关闭影响。
3. **Given** 某餐次尚未排菜单，**When** 商家查看首页，**Then** 对应餐次仍以近似稳定高度展示“未排菜单”和“先排菜单”入口，而不是消失。
4. **Given** 单个餐次订单汇总加载失败，**When** 页面完成加载，**Then** 该餐次的菜单与状态保留，仅汇总区域显示错误和重新加载入口。
5. **Given** 页面已有同一天旧数据，**When** 商家返回页面或触发刷新，**Then** 旧卡片继续可见并出现轻量刷新提示；较旧请求不得覆盖较新结果。
6. **Given** 商家点击待确认提醒、餐次非按钮区域或快捷入口，**When** 导航发生，**Then** 进入既有的订单、菜单、预订配置或配送清单页面；点击“手动加单”只进入加单页且不会同时触发卡片导航。
7. **Given** 会话失效或整页数据加载失败，**When** 商家进入首页，**Then** 分别沿用既有登录跳转，或看到明确错误和可用的重试入口。

### Edge Cases

- 待确认数量为零时不显示提醒占位；任一餐次汇总未知时，提醒必须明确数量是“已知”部分。
- 截止时间为空时显示“未设置截止时间”；价格为空时显示“商家默认价”，不得虚构金额或显示 `¥0` 代替未知价格。
- 菜单菜品较多或名称较长时，摘要最多两行且不撑破卡片；菜品构成摘要只根据真实菜单项计算。
- 顾客预订已截止、餐次关闭或菜单尚未开放时，商家对既有餐次仍可手动补录订单。
- 今日没有任何餐次时，午餐和晚餐空状态都存在，配送清单回退到订单页的安全默认状态。
- 小屏设备可以纵向滚动，固定底部导航不得遮挡最后一个快捷入口，并兼容底部安全区。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 页面 MUST 按品牌栏、问候区、条件式待确认提醒、午餐卡片、晚餐卡片、四个快捷入口和固定底部导航的顺序展示。
- **FR-002**: 问候 MUST 使用当前商家名称并根据当前时间显示自然中文问候；问候区不得展示头像、桃子符号、图片、装饰圆形或伪造的微信系统胶囊。
- **FR-003**: 待确认提醒 MUST 只在已知待确认数量大于零时出现，使用“有 N 笔订单待确认”语义；存在局部汇总失败时 MUST 明确这是已知数量。
- **FR-004**: 午餐和晚餐 MUST 始终各有一个卡片；既有餐次显示真实菜单、状态、已确认份数、价格、截止时间和汇总，缺失餐次显示“未排菜单”空状态。
- **FR-005**: 餐次状态 MUST 区分预订中、菜单已排好/待开放、已截止、已关闭和未排菜单，并使用一致且可读的状态文案与视觉标识。
- **FR-006**: 菜单摘要 MUST 来自真实菜单项，并在可确定时补充真实的菜、汤构成；长摘要 MUST 限制为两行。
- **FR-007**: 订单汇总成功时 MUST 突出显示真实已确认份数；局部失败时 MUST 保留餐次其他信息并提供局部重新加载。
- **FR-008**: 每个既有餐次 MUST 保留至少 88rpx 高的“手动加单”触控入口，并携带当天日期和对应午/晚餐进入既有手动加单页。
- **FR-009**: 餐次卡片非按钮区域 MUST 进入当天对应餐次订单；“手动加单”、局部重试、先排菜单和去开放操作 MUST 阻止卡片重复导航。
- **FR-010**: 菜单已排好但预订未开放的餐次 MUST 显示明确说明、真实价格或“商家默认价”，并提供“去开放”入口；不得虚构金额或截止时间。
- **FR-011**: 快捷入口 MUST 恰好为“排本周菜单”“开放预订”“查看订单”“配送清单”，并复用既有页面；配送清单优先定位今日有待配送的餐次，否则定位任一今日餐次，再否则安全回退订单页。
- **FR-012**: 固定底部导航 MUST 复用现有四栏“今日 / 菜品 / 菜单 / 订单”，突出当前“今日”项并兼容底部安全区；不得新增“我的”栏目。
- **FR-013**: 页面 MUST 保留首次加载、保留旧数据刷新、整页失败重试、单餐次汇总失败重试、快速重复刷新竞态保护和认证失效处理。
- **FR-014**: 页面视觉 MUST 使用温暖奶油背景、深褐主文字、砖红品牌色、绿色/金色/灰米色状态、统一圆角和轻阴影；不得使用通用 SaaS 后台风格、蓝紫渐变、玻璃拟态或强阴影。
- **FR-015**: 页面 MUST 在 H5 与微信小程序端使用相同真实数据与交互能力，不得增加 mock、静态 JSON 或仅 H5 生效的业务分支。
- **FR-016**: 实现 MUST NOT 修改后端接口、共享数据契约、头像能力、聊天、AI、在线支付或未存在的页面。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在 706×1572 参考视口中，品牌、问候、提醒、两张餐次卡片、四个快捷入口和四栏底部导航均按目标顺序出现，横向溢出为 0，最后一个快捷入口不被导航遮挡。
- **SC-002**: 对午晚餐五类状态、空餐次、零/非零待确认、空价格、空截止时间、局部失败、整页失败和保留数据刷新场景，规定文案和操作呈现覆盖率为 100%。
- **SC-003**: 待确认提醒、两张餐次卡片、四个快捷入口和四栏底部导航的所有既有目标跳转自动化通过率为 100%，手动加单触发重复导航次数为 0。
- **SC-004**: 既有商家 session、业务日期、真实菜单/订单汇总、局部失败隔离、请求竞态保护和认证失效相关自动化回归通过率为 100%。
- **SC-005**: H5 自动化测试与 H5、微信小程序生产构建全部通过，且相关单元测试维持项目要求的 100% 覆盖率。

## Assumptions

- 本功能以指定的 v0.2 高保真参考图为视觉基准，以完整还原说明为业务边界；二者冲突时，以真实产品能力和状态语义为准。
- 现有商家订单页支持通过日期和餐次定位；现有菜单页与预订批次页继续承载“排菜单”和“开放预订”。
- 微信右上角胶囊由系统负责；H5 不绘制对应占位或伪控件。
- 设计图和 HTML 源不需要随实现 PR 一并提交，后续如归档将使用独立设计资产 PR。
