# 功能规格：商家菜品库高保真重构

**功能分支**：`codex/kith-inn-v1-merchant-offerings-hifi`

**创建日期**：2026-07-24

**状态**：Draft

**输入**：基于现有菜品 CRUD、启停和批量导入能力，按 Page 2 高保真参考设计重构商家菜品库；保留真实业务行为、认证和错误处理，并修复旧响应覆盖新意图等竞态。

## 用户场景与测试

### User Story 1 - 快速浏览和筛选可排菜单的菜品（Priority: P1）

商家打开菜品库后，能够一眼看到当前启用的常做菜，按全部、荤菜、素菜和汤筛选，并从列表直接启停某一道菜。

**优先级原因**：浏览与启停是排菜单前最高频的准备动作，也是 Page 2 的核心价值。

**独立测试**：使用同时包含三类菜品和停用菜品的商家账号，验证默认列表只展示启用菜品、四个筛选结果准确，并验证启停仅影响目标菜品。

**验收场景**：

1. **Given** 商家拥有多类启用和停用菜品，**When** 打开菜品库，**Then** 默认选中“全部”且只展示启用菜品。
2. **Given** 当前列表含三类菜品，**When** 依次选择荤菜、素菜和汤，**Then** 每次只显示当前分类并保留筛选选择。
3. **Given** 当前菜品为启用状态，**When** 商家关闭该菜品，**Then** 仅该菜品进入停用状态，旧请求返回不得覆盖之后的新意图。
4. **Given** 列表正在加载、为空、筛选结果为空或加载失败，**When** 页面渲染，**Then** 商家看到对应且可恢复的明确状态。

---

### User Story 2 - 在管理模式维护菜品（Priority: P1）

商家进入管理模式后，可以查看启用中和已停用分组，新增、编辑或恢复菜品，并在保存后保持原有列表顺序和当前浏览上下文。

**优先级原因**：菜品资料正确性直接影响后续菜单生成，编辑结果不能让用户失去目标位置。

**独立测试**：进入管理模式，对列表中间的一道菜执行编辑，对新菜执行新增，再启停不同菜品；验证分组、顺序、筛选和错误恢复均符合预期。

**验收场景**：

1. **Given** 商家进入管理模式，**When** 页面完成刷新，**Then** 菜品按启用中和已停用分组展示，且不会因刷新丢失已有可用列表。
2. **Given** 商家编辑列表中间的菜品，**When** 保存成功，**Then** 目标菜品在原位置更新，其他菜品顺序不变。
3. **Given** 商家填写合法菜名和分类，**When** 新增成功，**Then** 新菜出现在列表中并给出成功反馈。
4. **Given** 保存、启停或刷新失败，**When** 请求结束，**Then** 页面保留可继续操作的上下文并给出清晰错误反馈。

---

### User Story 3 - 安全地批量导入菜品（Priority: P1）

商家在管理模式打开批量导入，粘贴文本后先查看预览，决定是否覆盖重名项，再确认导入；编辑原文后任何旧预览、旧冲突选择或旧结果都必须失效，确认提交后则必须锁定本次原文直至请求结束。

**优先级原因**：冲突选择按行生效，陈旧预览可能导致覆盖错误菜品，属于数据正确性风险。

**独立测试**：让两个预览请求以相反顺序返回，在预览期间修改原文，并在提交期间尝试修改原文；验证只能提交与当前文本对应的最新预览和冲突选择，提交中的原文不可变更。

**验收场景**：

1. **Given** 商家输入包含新增、重名和非法行的文本，**When** 请求预览，**Then** 每行结果与汇总清晰可辨，默认不覆盖重名菜品。
2. **Given** 预览请求尚未返回，**When** 商家修改原文并发起新预览，**Then** 旧响应不能恢复已失效的预览或冲突选择。
3. **Given** 当前文本与已显示预览不一致，**When** 商家尝试确认导入，**Then** 系统拒绝使用陈旧预览提交。
4. **Given** 最新预览有效，**When** 商家确认导入，**Then** 原文与冲突选择锁定到请求结束，仅按确认时的快照写入，并展示与本次提交对应的结果。

---

### User Story 4 - 在移动端获得稳定一致的高保真界面（Priority: P2）

商家在 H5 和微信小程序中都能使用与参考设计一致的菜品库：标题、筛选、菜品卡、开关、固定新增按钮、底部导航和编辑弹层在窄屏下不遮挡、不溢出且文字可读。

**优先级原因**：高保真还原是本次重构目标，但必须建立在前三个场景的数据正确性之上。

**独立测试**：在 354×786 视口核对默认态、管理态和弹层，并构建 H5 与微信小程序产物。

**验收场景**：

1. **Given** 354×786 的移动端视口，**When** 商家浏览默认列表，**Then** 首屏层级、颜色、间距和固定区域与参考图一致，列表内容不被操作区永久遮挡。
2. **Given** 菜名或主料较长，**When** 进入管理模式，**Then** 文本按设计约束截断或换行，不挤压编辑按钮和开关。
3. **Given** 操作暂不可用，**When** 按钮处于 disabled 状态，**Then** 文案仍可辨认且不可被误认为空白控件。
4. **Given** 相同业务数据，**When** 分别在 H5 和微信小程序打开页面，**Then** 核心信息、操作入口和状态反馈保持一致。

### 边界情况

- 商家没有任何菜品、当前分类没有启用菜品或仅有停用菜品。
- 两道不同菜品同时启停，同一道菜被快速重复操作，响应乱序返回。
- 编辑列表中间的菜品、编辑后分类变化、保存期间再次输入或关闭弹层。
- 导入原文在预览中被修改、提交中尝试修改，多个预览响应乱序返回，重名行号随文本变化。
- 认证过期、会员失效、CMS/后端超时或业务请求失败。
- 超长菜名、超长主料、50 行合法导入和含非法行的混合导入。
- 刘海屏安全区、窄屏和长列表下固定新增按钮与底部导航的叠放。

## 需求

### 功能需求

- **FR-001**：系统必须默认展示当前商家的启用菜品，并提供全部、荤菜、素菜和汤四个固定筛选项。
- **FR-002**：系统必须为加载中、空列表、筛选空结果、加载失败和静默刷新失败提供互不混淆的状态反馈。
- **FR-003**：商家必须能从菜品卡直接启停目标菜品；处理中只锁定目标菜品，不阻塞其他菜品。
- **FR-004**：同一菜品的旧请求结果不得覆盖该菜品之后产生的新操作意图。
- **FR-005**：管理模式必须按启用中和已停用分组展示全部菜品，并提供新增、编辑、恢复和批量导入入口。
- **FR-006**：编辑成功必须原位替换目标菜品，不改变其他菜品的相对顺序或当前筛选状态；新增成功可以追加新菜品。
- **FR-007**：保存、启停、刷新和导入失败不得伪造成功状态，认证失败必须沿用现有登录跳转。
- **FR-008**：任何导入原文变化都必须立即清除旧预览、旧冲突选择和旧提交结果。
- **FR-009**：预览或提交请求必须绑定发起时的原文版本；过期响应不得重新写入当前界面状态。
- **FR-010**：确认导入必须仅使用与当前原文匹配的最新预览及冲突选择，默认跳过重名菜品；提交请求进行中必须锁定原文和冲突选择，直至请求结束。
- **FR-011**：页面必须保留现有 API、数据模型、权限边界和真实 CRUD/导入能力，不新增伪造字段、图片或后端契约。
- **FR-012**：移动端界面必须覆盖默认浏览、管理、加载、空态、错误、保存中、启停中和导入中的视觉状态。
- **FR-013**：长文本、disabled 控件、固定新增按钮和底部导航在 354×786 视口下必须可读、可点且互不永久遮挡。
- **FR-014**：相同功能必须能在 H5 与微信小程序构建中使用。
- **FR-015**：实现必须有自动化测试覆盖筛选/分组纯逻辑、编辑顺序、并发启停、陈旧导入响应和主流程回归。

### 关键实体

- **菜品**：商家可重复使用的菜品资料，包含名称、可选主料、分类、启用状态和稳定标识。
- **菜品列表上下文**：当前浏览或管理模式、分类筛选、菜品顺序、加载状态及逐菜品进行中的操作。
- **导入草稿版本**：一次原文快照及其版本标识，用于绑定预览、冲突选择和提交结果。
- **导入预览**：针对某个导入草稿版本生成的逐行结果与汇总；只有与当前版本匹配时才可提交。

## 成功标准

### 可衡量结果

- **SC-001**：商家在打开页面后 3 次以内点击即可完成任一分类筛选或单菜品启停。
- **SC-002**：覆盖至少 2 个菜品的乱序启停测试中，最终状态 100% 与每道菜最后一次用户意图一致。
- **SC-003**：所有编辑回归场景中，目标菜品以外的相对顺序 100% 保持不变。
- **SC-004**：覆盖原文修改和乱序响应的导入测试中，陈旧预览、冲突选择和结果 0 次重新进入当前状态，陈旧预览 0 次可提交，提交 pending 期间原文 0 次可变更。
- **SC-005**：默认态、管理态、新增/编辑弹层和导入态在 354×786 视口下无不可读按钮、横向溢出或固定区域永久遮挡关键操作。
- **SC-006**：H5 与微信小程序均能完成构建，现有商家菜品与接龙主流程自动化测试全部通过。

## 假设

- 继续复用现有商家认证、菜品 API、批量导入协议和底部导航，不修改后端或 CMS。
- 高保真参考图定义默认浏览态；管理态和导入态在同一视觉语言下补齐，并优先保证真实数据可用性。
- 菜品列表沿用服务返回顺序；本功能不引入手工排序。
- 设计 Prompt 仅作为本地输入，不进入任何提交；Page 2 参考 PNG 与共享 HTML 设计源必须通过独立文档 PR 入库，并在高保真视觉切片开始前合并，使验收不依赖本地文件。
