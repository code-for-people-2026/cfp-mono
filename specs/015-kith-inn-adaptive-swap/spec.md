## 用户场景与测试

### User Story 1 - 小菜品池仍能自动换菜（Priority: P1）

作为桃子，我在菜单里点某道菜的“换”，只要启用菜品池里还有当前餐没用过的同类菜，系统就替我选出冲突最少的一道，而不是因为近期做过或主料重复而直接失败。

**Why this priority**: 自动换菜是高频操作；当前硬过滤会把实际可用的菜误判成“没有替代品”，直接阻断核心菜单流程。

**Independent Test**: 分别用充足、小池、仅一个可替换候选、完全无候选四种菜品池执行自动换菜；前三种成功且选择符合固定优先级，最后一种才失败。

**Acceptance Scenarios**:

1. **Given** 同类候选中存在不违反任何避重偏好的菜，**When** 桃子自动换菜，**Then** 系统从零冲突的最优候选中选择，且不报告放宽规则。
2. **Given** 所有同类候选都至少违反一项避重偏好，**When** 桃子自动换菜，**Then** 系统仍选择按固定优先级冲突最少的候选，并报告实际放宽的规则。
3. **Given** 只有一个启用、同类且当前餐未使用的候选，**When** 桃子自动换菜，**Then** 无论它是否近期出现或与本餐主料重复，系统都完成换菜并报告相应放宽规则。
4. **Given** 没有任何启用、同类且当前餐未使用的候选，**When** 桃子自动换菜，**Then** 系统返回“没有可替换菜品”，且菜单不变。

---

### User Story 2 - 看见本次放宽原因（Priority: P1）

作为桃子，我在菜品池较小时完成自动换菜后，能在菜单页看到这次允许了哪些重复，知道系统没有静默忽略自己的避重偏好。

**Why this priority**: 放宽规则是自适应策略的信任边界；没有可见说明，成功换菜也可能让用户误以为避重逻辑失效。

**Independent Test**: 在 H5 菜单页连接小菜品池测试链路，点击“换”后确认菜名更新，并显示与响应规则一致的中文放宽提示。

**Acceptance Scenarios**:

1. **Given** 自动换菜没有放宽规则，**When** 换菜成功，**Then** 菜单更新且不显示误导性的放宽提示。
2. **Given** 自动换菜放宽了一项或多项规则，**When** 换菜成功，**Then** 菜单更新并显示“菜品池较小”及可读的具体放宽原因。
3. **Given** 换菜响应包含多项放宽规则，**When** 菜单页提示，**Then** 原因按固定规则优先级展示，不遗漏也不重复。

---

### User Story 3 - 原有菜单编辑保护不回归（Priority: P1）

作为桃子，我仍可点名指定替代菜，即使它有冲突也以我的选择为准并给 warning；已发布菜单仍必须确认后才能修改，修改后旧发布文案失效。

**Why this priority**: 自适应只属于自动选择分支，不能改变用户显式决定或已发布菜单的安全边界。

**Independent Test**: 对同一菜单分别执行有冲突的指定换菜、无 force 的 published 换菜和带 force 的 published 换菜；验证 warning、409 和清空旧文案的既有行为不变。

**Acceptance Scenarios**:

1. **Given** 桃子指定了一道池内替代菜且它违反主料避重，**When** 执行指定换菜，**Then** 系统保留她的选择并返回既有冲突 warning，不改用自动评分候选。
2. **Given** 菜单已发布且未确认 force，**When** 发起自动或指定换菜，**Then** 系统拒绝修改。
3. **Given** 菜单已发布且桃子确认 force，**When** 换菜成功，**Then** 仅目标菜被替换，其他菜及顺序不变，旧发布文案被清空。

### Edge Cases

- 目标餐不存在或目标菜不在该餐时，维持既有错误语义，不读取或修改菜单。
- 停用菜、不同分类菜、目标菜本身以及当前餐已使用的菜绝不进入自动候选。
- 候选没有主料时，只计算同一道菜相关冲突，不虚构主料冲突。
- 当前目标餐可能同时出现在历史查询结果中；评分前必须按 plan 身份排除，避免重复计罚，但当前餐未替换的菜仍参与同日主料冲突。
- 目标日之前第 1–7 个日历日属于“近 7 日”；目标日、第 8 日及未来日期不属于该窗口。
- 同周按目标日期所在的周一至周日判断；相邻月份或年份不改变自然周边界。
- 多个候选四级冲突计数完全相同时，只在这些并列候选中随机；固定随机源必须得到稳定结果。
- 随机源返回边界值时不得越界或选择非并列候选。
- 自动换菜成功后必须只替换目标位置，不能重排或改变同餐其他菜。

## Requirements

### Functional Requirements

- **FR-001**: 自动换菜候选 MUST 仅包含启用、与目标菜同分类、非目标菜且当前餐尚未使用的菜。
- **FR-002**: 系统 MUST 为每个候选计算以下四级冲突计数，并严格按顺序做字典序比较：同周同菜、同日同主料、目标日前 1–7 日同菜、目标日前 1–7 日同主料。
- **FR-003**: 同日同主料计数 MUST 同时包含目标日其他历史餐次和当前餐未被替换的菜；当前目标餐自身 MUST 从历史集合排除。
- **FR-004**: 系统 MUST 从最小四级冲突计数的候选中选择；不得为了满足较低优先级规则而接受更高优先级的额外冲突。
- **FR-005**: 只有候选四级冲突计数完全相同才允许随机决胜；随机源 MUST 可注入，以便稳定验证。
- **FR-006**: 自动换菜成功 MUST 返回按固定顺序排列的 `relaxedRules`；候选在某级冲突计数大于 0 时才包含对应规则。
- **FR-007**: 放宽规则标识 MUST 固定为 `same-week-offering`、`same-day-main-ingredient`、`recent-offering`、`recent-main-ingredient`，顺序与 FR-002 一致。
- **FR-008**: 系统 MUST 使用与目标日期相关的菜单历史；查询范围至少覆盖目标日前 7 日和目标所在自然周，并按 plan 身份排除当前目标餐。
- **FR-009**: 只要存在一个符合 FR-001 的候选，自动换菜 MUST 成功；只有候选集合为空才返回 `no-alternative` / “没有可替换菜品”。
- **FR-010**: 自动换菜 MUST 只替换目标菜所在位置，其他菜及顺序保持不变。
- **FR-011**: 菜单页 MUST 在 `relaxedRules` 非空时显示“菜品池较小”及每项规则的中文原因；空数组时不得显示放宽提示。
- **FR-012**: H5 E2E MUST 覆盖小菜品池仍能换菜及放宽提示，并为后续 #157 的真实链路 E2E 保留可扩展入口。
- **FR-013**: 用户指定换菜 MUST 保持“用户选择优先 + 既有冲突 warning”，不得进入自动评分流程。
- **FR-014**: published 菜单无 force 拒绝、确认后二次修改并清空旧 `publishText` 的行为 MUST 保持不变。
- **FR-015**: 本功能 MUST NOT 修改 kith-inn-v1、菜单生成算法、CMS schema，不得引入 LLM、推荐系统或仓库尚不存在的第三方库/版本；允许 FE package 声明仓库已锁定的 Playwright 版本以运行 H5 E2E。

### Key Entities

- **自动换菜候选**: 启用菜品池中符合分类和当前餐排除条件的一道菜；拥有菜 ID、分类与可选主料。
- **换菜评分**: 候选在四级避重规则上的冲突计数元组；按固定优先级比较。
- **放宽规则（Relaxed Rule）**: 胜出候选实际发生冲突的规则标识，供 API 和菜单页解释。
- **菜单历史**: 与目标日期相关、排除当前目标 plan 的已存菜单餐次；用于同周、同日与近 7 日计数。

## Success Criteria

### Measurable Outcomes

- **SC-001**: 充足、小池、仅一个可替换候选、完全无候选四类自动换菜边界均有稳定自动化测试，预期通过率 100%。
- **SC-002**: 对任意存在合格候选的测试样本，自动换菜成功率为 100%；仅候选数为 0 时失败。
- **SC-003**: 固定历史和随机源下，胜出候选与 `relaxedRules` 的测试结果可重复率为 100%。
- **SC-004**: 每次成功换菜除目标位置外的菜单差异数为 0，目标位置差异数为 1。
- **SC-005**: H5 小池场景中，换菜结果和放宽中文提示均可见，端到端验收通过率为 100%。
- **SC-006**: 指定换菜 warning、published 无 force 拒绝和 force 后清空旧文案的既有回归测试通过率为 100%。
- **SC-007**: `pnpm verify` 全仓质量门禁通过。

## Assumptions

- 已有菜单 plan 的 `date` 使用 `YYYY-MM-DD`；自然周以周一为首日，日期比较按日历日，不受运行机器时区影响。
- 历史只采用当前 seller 通过既有 CMS 租户边界返回的 draft/published 菜单；不改变租户隔离与状态模型。
- `relaxedRules` 是本次自动换菜的瞬时解释，不持久化到 CMS；刷新后的菜单仍保留菜品结果，但不重放旧提示。
- H5 E2E 在本功能内只建立和验证菜单换菜所需的最小设施；#157 后续扩展为完整 H5 → BE → CMS → PostgreSQL 与跨租户路线级覆盖。
- 该功能触发跨 shared/BE/FE、API 响应契约与四段 PR，按宪法采用全套 spec。
