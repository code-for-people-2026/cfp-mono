# Feature Specification: 街坊味 v1 顾客预订登记

**Feature Branch**: `codex/kith-inn-v1-m2-spec`

**Created**: 2026-07-11

**Status**: Draft

**Input**: User description: "完成 M1 后进入 M2；桃子创建并分享预订批次，顾客从微信小程序卡片静默识别身份，选择或新增称呼与地址资料，为批次内多个餐次登记份数，并在桃子确认前查看、修改或取消自己的草稿订单。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 创建并分享预订批次 (Priority: P1)

作为桃子，我可以配置餐次的价格、截止时间和开放状态，选择若干可预订餐次生成一张分享卡片，从而把本周可登记的午晚餐一次发到微信群。

**Why this priority**: 顾客入口必须从一个真实、受控的预订批次解析商家和可选餐次；这是后续顾客身份与订单能力的业务入口。

**Independent Test**: 使用桃子商家会话配置一个未来餐次为开放，创建包含该餐次的预订批次，验证分享标题和公开标识；关闭批次或餐次后，分享入口仍可查看但不能新增登记。

**Acceptance Scenarios**:

1. **Given** 一个已有完整菜单的未来餐次，**When** 桃子设置未来截止时间、价格并开放预订，**Then** 该餐次可被选择进预订批次。
2. **Given** 桃子选中一个至二十个属于当前商家且仍开放、未截止的餐次，**When** 创建预订批次，**Then** 系统保存选择、生成不可顺序猜测的公开标识和清晰分享标题。
3. **Given** 某餐次属于其他商家、未开放、已截止或已关闭，**When** 桃子尝试把它加入批次，**Then** 系统拒绝整个创建请求且不泄露其他商家数据。
4. **Given** 一个已创建的批次，**When** 桃子使用微信转发能力分享，**Then** 卡片路径只包含公开标识，不包含数据库标识、商家标识或身份信息。
5. **Given** 一个开放批次，**When** 桃子关闭该批次，**Then** 只有这张分享入口停止接受新登记，关联餐次、其他批次和已有订单不被改变。
6. **Given** 一个餐次被多个批次引用，**When** 桃子关闭该餐次，**Then** 所有引用它的入口都停止接受该餐次的新登记，但仍可读取已有订单。

---

### User Story 2 - 从分享卡片静默进入 (Priority: P2)

作为顾客，我点击微信群里的预订卡片后，不需要注册、手机号或头像昵称授权，就能看到这张卡片里的商家、餐次、菜单、价格和可登记状态。

**Why this priority**: 这是顾客侧的最小可用入口，先证明公开批次解析、轻身份和商家隔离，再开放个人资料和订单写入。

**Independent Test**: 使用有效、关闭和无效的批次公开标识分别进入；验证有效批次静默建立顾客会话并展示限定餐次，关闭批次只读展示，无效标识和身份交换失败均不能建立会话。

**Acceptance Scenarios**:

1. **Given** 一个有效批次公开标识和成功的微信身份交换，**When** 顾客打开卡片，**Then** 系统建立只代表该微信身份与批次所属商家的顾客会话，不显示任何登录或注册步骤。
2. **Given** 顾客同时是该商家的 operator，**When** 从分享卡片进入，**Then** 当前入口仍是顾客会话且不展示商家操作。
3. **Given** 一个开放批次，**When** 顾客查看页面，**Then** 只展示该批次所含餐次，并为每个餐次展示菜单、最终单价、截止时间和当前是否可登记。
4. **Given** 一个 closed 或 archived 批次，**When** 顾客打开卡片，**Then** 仍可建立会话并查看批次和自己的历史订单，但页面明确禁止新增登记。
5. **Given** 批次公开标识不存在，**When** 顾客打开卡片，**Then** 页面显示“这个预订登记链接已失效”，不建立顾客会话，也不暴露任何商家数据。
6. **Given** 微信身份交换失败、超时或返回空身份，**When** 顾客进入，**Then** 系统提示重新进入小程序且不得回退为匿名写入或开发身份。

---

### User Story 3 - 选择资料并提交预订登记 (Priority: P3)

作为顾客，我可以选择以前保存的一整条“称呼 + 地址”资料，或新建资料，然后为批次内一个或多个餐次填写份数并确认总价，从而完成预订登记。

**Why this priority**: 这是 M2 的核心交易价值；它建立在已验证的顾客会话和可分享批次上，并把登记写入 M1 已有订单闭环。

**Independent Test**: 以首次顾客和已有多资料顾客分别进入开放批次，完成资料选择/新建、两个餐次份数确认与提交；重复提交更新原订单，部分餐次状态变化时逐项返回且可安全重试。

**Acceptance Scenarios**:

1. **Given** 顾客没有有效历史资料，**When** 开始登记，**Then** 必须填写称呼和地址，并看到“用于桃子识别订单和送餐地址”的用途说明。
2. **Given** 顾客只有一条有效历史资料，**When** 进入登记页，**Then** 系统默认选中整条资料；有多条时必须由顾客明确选择，不拆分称呼和地址自由组合。
3. **Given** 顾客修改本次使用的称呼或地址，**When** 不选择保存，**Then** 只写入订单快照；选择保存时新建一条资料，不静默覆盖原资料。
4. **Given** 批次包含多个可登记餐次，**When** 顾客为至少一个餐次填写正整数份数，**Then** 提交前展示称呼、地址、逐餐份数、单价和总价确认摘要。
5. **Given** 顾客确认提交，**When** 所有目标仍在开放批次内、餐次仍开放且未截止，**Then** 系统为每个目标创建或更新一条草稿订单，来源为顾客卡片，并保存顾客、价格和地址快照。
6. **Given** 同一资料和餐次已有草稿订单，**When** 顾客再次提交，**Then** 系统更新同一订单而不是新增；已有取消订单时，必须明确确认后才重置同一记录为草稿。
7. **Given** 多餐次提交期间其中一个餐次已关闭或过期，**When** 系统处理请求，**Then** 合法餐次独立成功、失败餐次逐项说明原因，顾客可安全重试且不会产生重复订单。
8. **Given** 顾客提交另一个微信、商家、批次外餐次或无 openid 的手工资料标识，**When** 系统校验归属，**Then** 必须拒绝且响应不包含目标资料或订单内容。

---

### User Story 4 - 查看并管理自己的登记 (Priority: P4)

作为顾客，我可以查看自己在当前商家下的订单状态，并在桃子确认前、餐次仍开放且未截止时修改份数或取消草稿，从而纠正登记错误。

**Why this priority**: 提交后的自助纠错和只读生命周期构成顾客闭环，但不能先于身份、批次和首次登记开放。

**Independent Test**: 创建 draft、confirmed、canceled 订单以及已截止/关闭餐次；验证顾客只看到当前微信订单，只能修改或取消仍可写的 draft，桃子确认后立即锁单，同时软停用资料不影响历史订单。

**Acceptance Scenarios**:

1. **Given** 当前微信在该商家有多个状态的订单，**When** 查看“我的预订”，**Then** 只显示 `customerOpenid` 属于当前微信的订单，并使用顾客可理解的业务、付款和送达文案。
2. **Given** 一个自己的草稿订单且餐次开放、未截止，**When** 顾客修改为新的正整数份数，**Then** 同一订单更新并在商家侧刷新后可见。
3. **Given** 一个自己的草稿订单且餐次仍可写，**When** 顾客二次确认取消，**Then** 同一订单变为已取消并保留历史。
4. **Given** 桃子已确认订单，**When** 顾客尝试修改或取消，**Then** 系统拒绝并提示“桃子已确认，请在群里联系桃子”。
5. **Given** 餐次已关闭或超过截止时间，**When** 顾客尝试修改或取消草稿，**Then** 系统拒绝并提示“本餐次已截止，请在群里联系桃子”。
6. **Given** 顾客选择删除一条资料，**When** 二次确认，**Then** 资料被软停用且后续不再出现在选择列表，历史订单快照和订单可见性不变。
7. **Given** 顾客提交其他微信或其他商家的 profile/order 标识，**When** 尝试读取、修改或取消，**Then** 系统统一表现为不存在且不得泄露目标记录内容。

### Edge Cases

- 同一批次提交完全相同的重复餐次项时只处理一次；同一餐次出现不同份数或不同重登记标志时拒绝整个请求，避免静默选择冲突值。一次最多处理二十个去重餐次。
- 批次创建或顾客提交期间，餐次/批次状态或截止时间发生并发变化时，以写入前最后一次服务端校验为准，不接受过期快照。
- 批次关闭不关闭餐次；餐次关闭会影响所有引用它的批次；两者都不自动取消已有订单。
- closed/archived 批次可以换取顾客会话并读取历史，但不能用于创建、重提、修改或取消订单。
- 顾客会话有效期间从另一个 seller 的分享入口进入时，必须为新 seller 建立独立会话，不能沿用旧 seller 权限。
- 顾客资料被软停用后不能用于新登记；订单仍按保存的 `customerOpenid` 对原顾客可见。
- 顾客创建 profile 成功但订单逐项写入失败时，profile 作为有效资料保留，不执行跨请求回滚。
- 同一坐标并发创建/更新订单时，最终只保留一条记录；另一请求得到可重试的冲突或更新结果。
- 分享标题超过限制、公开标识冲突或随机源失败时，不创建残缺批次。
- 顾客侧任何错误响应都不得包含 openid、operator 身份、数据库关系标识或其他顾客的个人信息。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在 M1 已有 v1 商家前端、业务服务、共享内容管理实例和七个 v1 collection 上交付 M2，不得增加新业务 collection、数据库、常驻内容管理进程或第二套顾客应用。
- **FR-002**: 桃子必须能够为当前商家的餐次设置最终价格、未来截止时间和 `draft/open/closed` 预订状态；只有菜单完整、状态为 open、截止时间仍在未来的餐次可接受顾客写入。
- **FR-003**: 桃子必须能够从当前商家选择一至二十个仍可登记的餐次创建预订批次；系统必须校验所有餐次和创建者属于当前商家。
- **FR-004**: 每个预订批次必须拥有不可顺序猜测且全局唯一的公开标识、清晰标题、状态、所选餐次和创建者；分享路径必须由固定顾客页面和公开标识派生，不得持久化或暴露内部标识。
- **FR-005**: 桃子必须能够关闭当前商家的批次或餐次；关闭批次只影响该分享入口，关闭餐次影响所有引用它的批次，二者都不得取消或改写已有订单。
- **FR-006**: 顾客必须通过分享卡片中的有效公开标识和微信临时身份静默建立与单一商家绑定的顾客会话；不得要求注册、手机号、头像、昵称或 `wx.getUserProfile` 授权。
- **FR-007**: 顾客会话必须与 operator、共享内容管理后台及旧产品身份分离，只代表当前商家和已验证微信身份，并具有明确过期时间；所有后续顾客读写必须从会话推导商家和微信身份。
- **FR-008**: 任意状态的现存批次都必须允许建立顾客会话和读取历史；只有 open 批次可以作为新登记或重登记入口，无效公开标识不得建立会话或泄露商家数据。
- **FR-009**: 顾客批次页面必须只公开该批次所含餐次的商家名、日期、餐次、菜单快照、最终单价、截止时间和可登记状态；不得公开 operator、其他批次或内部关系标识。
- **FR-010**: 系统必须支持顾客列出、新增和软停用当前商家下绑定自己微信身份的资料；不得展示或写入其他微信、无 openid 手工资料或其他商家资料。
- **FR-011**: 每条顾客资料必须是不可拆分的“称呼 + 地址”；本次修改可以只进入订单快照，选择保存时必须创建新资料，不得依据称呼或地址自动认领、合并或覆盖历史资料。
- **FR-012**: 顾客提交前必须至少选择一个可登记餐次并填写正整数份数，且看到资料、逐餐份数、单价和总价确认摘要；一次最多处理二十个去重餐次。
- **FR-013**: 顾客提交必须逐项重新验证批次 open、目标餐次属于该批次、餐次 open、截止时间未到、profile active 且 `profile.openid` 等于当前微信身份；不得信任客户端提交的商家、openid、价格或订单状态。
- **FR-014**: 顾客卡片订单必须保存顾客资料、已验证 customerOpenid、顾客/地址/单价快照、正整数份数、`customer-card` 来源和 draft 初始状态；最终单价取餐次价格，为空时回退商家默认价格。
- **FR-015**: 同一商家、餐次、顾客资料最终只能有一条订单；重复提交必须更新现有 draft，取消后重登记必须显式确认并复用同一记录，confirmed 订单不得由顾客提交覆盖。
- **FR-016**: 多餐次提交必须逐项返回 created/updated/resubmitted/failed 结果；单项失败不得回滚已成功项，重试不得制造重复订单。
- **FR-017**: 系统必须支持顾客按当前商家和 customerOpenid 查看自己的订单及三条状态轴，不得按称呼、地址或 profile 猜测订单归属。
- **FR-018**: 顾客只能修改或取消自己的 draft 订单，且写入时批次仍 open、餐次仍 open、截止时间未到；confirmed 或 canceled 订单的修改/取消，以及 closed/archived batch、closed slot 或过期入口的所有顾客写入必须拒绝。canceled 订单的显式重登记仅按 FR-015 处理。
- **FR-019**: 顾客取消 draft 必须二次确认并保留同一订单历史；顾客不得确认订单、修改付款/送达状态、编辑 confirmed 订单或操作商家订单能力。
- **FR-020**: 桃子在 M1 商家侧确认、取消、收款或送达订单后，顾客刷新必须看到一致只读状态；confirmed 锁单必须立即阻止顾客后续修改或取消。
- **FR-021**: 顾客软停用资料不得删除或改写历史订单快照，也不得使原顾客失去按 customerOpenid 查看历史订单的能力。
- **FR-022**: 所有批次、餐次、profile 和 order 读取、关系引用及状态变更必须同时校验 seller 和当前角色 owner；跨商家/跨微信统一按不存在处理且不得泄露目标记录内容。
- **FR-023**: 顾客首次填写资料时必须展示称呼与地址用途短文案；正式隐私指引配置、类目/资质验证和公开文案总审计留给 M3/M4。
- **FR-024**: M2 不得接在线支付、退款、配送路线、顾客头像昵称手机号、AI/agent 或接龙导入，也不得创建顾客自助认领/合并无 openid 手工资料的入口。
- **FR-025**: 本功能必须先补会失败的共享契约、身份隔离、tenant/owner、状态门禁、并发幂等和关键双端流程测试，再实现对应行为，并通过每个实现 PR 的窄验证与仓库完整质量门禁。

### Key Entities

- **预订餐次**: 当前商家某个日历日午餐或晚餐的菜单快照、最终价格、截止时间和开放状态；关闭后保留历史。
- **预订批次**: 桃子一次分享选择的一至二十个餐次，包含公开标识、标题、状态和创建者；关闭只影响该入口。
- **顾客会话**: 由有效批次和微信临时身份建立，只代表一个商家和一个微信身份，与 operator 会话隔离。
- **顾客资料**: 当前商家下绑定某微信的一整条称呼与地址，可软停用；无 openid 的商家手工资料理论上存在但不属于顾客侧范围。
- **顾客订单**: 某顾客资料在某餐次的份数登记，保存微信、顾客、地址和单价快照，并复用 M1 已有业务、付款、送达三条状态轴。

### Scope Boundaries

- M2 不修改菜品池或菜单生成规则，只增加餐次预订配置和分享引用。
- M2 不提供在线支付、退款、配送地图、路线规划或消息推送。
- M2 不采集手机号、微信昵称、头像或其他社交资料。
- M2 不做无 openid 手工资料的顾客认领、合并或自动匹配。
- M2 不实现微信群接龙导入、AI 辅助或审核兜底；这些属于 M3。
- M2 不建立正式 migration baseline、真实数据发布或类目/资质提交；这些属于 M4。
- M2 不新增 collection、订单明细、履约、审计、session 或 batch-slot 业务表。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 桃子可以在 60 秒内完成“配置餐次 → 选择餐次 → 创建可分享批次”，且分享入口中内部数据库标识暴露数量为 0。
- **SC-002**: 顾客从有效分享卡片进入后，在正常网络下 5 秒内看到商家和批次餐次；无需完成任何显式注册或个人资料授权步骤。
- **SC-003**: 首次顾客可在 90 秒内完成“填写称呼地址 → 选择两个餐次份数 → 确认摘要 → 提交”，已有单资料顾客可在 45 秒内完成同一流程。
- **SC-004**: 自动化验证中，无效批次、跨商家关系、跨微信 profile/order 和无 openid 手工资料的未授权读写必须 100% 被拒绝，响应中的其他顾客个人信息泄露数量为 0。
- **SC-005**: 对同一资料和餐次连续或并发提交两次后，符合唯一坐标的订单数量始终为 1；重试不会增加重复记录。
- **SC-006**: 批次关闭、餐次关闭或截止时间到达后，对应入口的新建、重登记、修改和取消成功数量均为 0，而已有订单可读率为 100%。
- **SC-007**: 桃子确认订单后，顾客下一次刷新即看到已确认且所有顾客修改/取消尝试均被拒绝；付款与送达状态也与商家侧一致只读展示。
- **SC-008**: 软停用资料后，该资料在后续选择列表中的出现数量为 0，相关历史订单和快照仍全部可读。
- **SC-009**: M2-A/B/C/D 四个切片可分别独立演示；完成任一切片后，M0/M1、旧项目入口和共享内容管理实例均通过回归验证。
- **SC-010**: M2 完成时仍只有七个 v1 业务 collection、一个 v1 前端、一个 v1 业务服务和共享内容管理进程，M3/M4 功能入口数量为 0。

## Assumptions

- MVP 生产范围仍只有桃子一个商家，但自动化测试至少覆盖两个 seller 和两个 customer openid。
- 所有 seller 使用同一个微信小程序 AppID；若未来增加 AppID，身份坐标必须先升级为 `appId + openid`。
- 顾客需要稳定网络；M2 不承诺离线登记。网络失败时保留当前表单并允许重试。
- 所有业务日期按 Asia/Shanghai 解释，截止时间使用明确时刻；是否过期只由服务端当前时间判定。
- 一次预订批次和一次顾客提交最多包含二十个去重餐次，足以覆盖当前一至两周午晚餐范围。
- 多餐次提交采用逐项结果和安全重试，不提供跨多个订单的全局事务。
- H5 只用于自动化和开发验证；真实分享行为与静默微信身份在微信小程序环境验证。
- 顾客会话可在现存批次关闭后继续用于读取当前商家下自己的历史订单，但任何订单写入仍必须携带并重新校验来源批次。
- M0 七个 collection 已包含 M2 所需字段和索引；若规划发现必须改变持久化契约，必须在本规格中记录原因、兼容性和独立迁移验证任务。
