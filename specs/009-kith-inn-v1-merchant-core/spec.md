# Feature Specification: 街坊味 v1 商家核心闭环

**Feature Branch**: `codex/kith-inn-v1-m1-spec`

**Created**: 2026-07-10

**Status**: Draft

**Input**: User description: "完成 M0 后进入 M1；先交付商家登录与菜品池，再交付菜单生成和商家侧订单闭环。创建真正可运行的 v1 前后端，不实现顾客预订、分享、AI 或支付。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 登录并维护菜品池 (Priority: P1)

作为桃子，我打开商家入口后可以用已经绑定的微信身份进入自己的经营空间，批量录入或手动维护菜品，从而得到可供菜单生成使用的干净菜品池。

**Why this priority**: 身份边界和菜品池是后续菜单、订单能力的共同入口；这个切片完成后，M1 首次具备可独立使用的商家功能。

**Independent Test**: 使用已绑定的桃子身份进入商家入口，完成菜品列表查看、单条新增/编辑/停用/恢复以及多行录入预览和提交；退出后用未绑定身份验证无法进入。

**Acceptance Scenarios**:

1. **Given** 桃子的微信身份已有一条有效商家成员资格且商家处于启用状态，**When** 她从商家入口进入，**Then** 系统直接选中唯一商家并只展示该商家的数据。
2. **Given** 同一微信身份拥有多个有效商家成员资格，**When** 她进入商家入口，**Then** 系统要求先选择本次经营的商家，且本次会话只能访问所选商家。
3. **Given** 微信身份未绑定有效商家成员资格，**When** 尝试进入商家入口，**Then** 系统拒绝进入且不自动创建商家或成员资格。
4. **Given** 桃子已进入自己的商家空间，**When** 新增或编辑菜品，**Then** 系统保存菜名、主料、分类和启用状态，并拒绝空菜名、无效分类或同商家重名。
5. **Given** 某菜品已用于历史菜单，**When** 桃子停用它，**Then** 它不再参与新菜单生成，但历史菜单快照保持可读。
6. **Given** 桃子粘贴多行菜品文本，**When** 查看解析预览，**Then** 系统逐行标出可导入、格式错误和重名冲突，提交前不写入任何数据。
7. **Given** 预览中存在同商家重名菜，**When** 桃子未主动选择覆盖，**Then** 默认跳过该行；选择覆盖时只更新该同名菜，不产生重复记录。

---

### User Story 2 - 生成并调整菜单 (Priority: P2)

作为桃子，我可以为一个餐次或一周的午晚餐生成菜单，并按需换掉其中一道菜，从而减少每天临时想菜单的工作。

**Why this priority**: 菜单把菜品池转化为真实经营计划，也是手动订单选择餐次的前置数据。

**Independent Test**: 在已准备足量启用菜品的商家空间中，生成一个餐次和一周菜单，验证每餐结构、快照、避重规则，再替换其中一道菜。

**Acceptance Scenarios**:

1. **Given** 启用菜品足以组成 2 荤、2 素、1 汤，**When** 桃子为一个日期和餐次生成菜单，**Then** 系统生成恰好五道互不重复的菜并保存当时的菜名、主料和分类快照。
2. **Given** 任一分类的启用菜品不足，**When** 桃子生成菜单，**Then** 系统明确指出缺少的分类和数量，不创建不完整菜单，也不虚构菜品。
3. **Given** 桃子选择一段日期及午餐/晚餐范围，**When** 批量生成，**Then** 系统为每个目标餐次生成菜单，在菜品池足够时避免同周重复同一道菜，并避免同一天重复同一主料。
4. **Given** 同一日期和餐次已有菜单，**When** 桃子再次生成，**Then** 系统先要求确认，确认后更新该餐次而不是创建重复餐次。
5. **Given** 桃子对一道菜不满意，**When** 选择换菜，**Then** 系统只替换目标菜，优先使用同分类、当前餐未出现且主料不冲突的启用菜品。
6. **Given** 没有符合条件的替代菜，**When** 桃子选择换菜，**Then** 原菜单保持不变并提示补充菜品池。

---

### User Story 3 - 手动记录并完成订单 (Priority: P3)

作为桃子，我可以为自己家或私聊顾客手动补单，查看每个餐次的订单并处理确认、收款、送达和取消，从而在没有顾客侧预订功能时也能跑通经营闭环。

**Why this priority**: 这是 M1 的最终交付价值；它不依赖分享卡片或顾客入口，能先验证菜单到履约的完整商家流程。

**Independent Test**: 为一个已有菜单的餐次创建无微信绑定的顾客资料和手动订单，依次完成修改、确认、标已付、标已送、回退付款/送达状态和取消，并核对汇总及纯文本清单。

**Acceptance Scenarios**:

1. **Given** 已有一个餐次，**When** 桃子手动补单，**Then** 她可选择已有顾客资料或新建“称呼 + 地址”资料，填写份数和备注，订单来源记录为手动补单，单价取餐次价格并在其为空时回退商家默认价格。
2. **Given** 私聊顾客尚未进入小程序，**When** 桃子新建资料，**Then** 微信身份可以为空，系统不得依据相同称呼或地址猜测绑定身份。
3. **Given** 同一顾客资料在同一餐次已有订单，**When** 桃子再次补单，**Then** 系统提示更新现有订单，不静默创建重复订单；现有订单已取消时，必须明确确认后才重置为草稿。
4. **Given** 一个草稿订单，**When** 桃子确认，**Then** 订单进入已确认状态并记录确认时间；已确认订单计入备餐、未付和未送汇总。
5. **Given** 一个未取消订单，**When** 桃子修改份数、地址快照或备注，**Then** 只修改当前订单；修改已确认订单前明确提示会影响备餐或配送。
6. **Given** 一个已确认订单，**When** 桃子标已付/未付或已送/待送，**Then** 两组状态互不影响，并同步维护对应时间。
7. **Given** 多个已确认订单，**When** 桃子批量标已送，**Then** 只更新她选中的当前商家订单。
8. **Given** 一个草稿或已确认订单，**When** 桃子取消，**Then** 订单保留历史并退出总份数、未付和未送口径；只有再次补同一顾客资料和餐次并明确确认时才能重置为草稿。
9. **Given** 某餐次包含已确认订单，**When** 桃子查看或复制清单，**Then** 清单仅统计 `confirmed` 订单，包含餐次、总份数、顾客、份数和地址，不计入 `draft` 或 `canceled` 订单。

### Edge Cases

- 微信身份交换失败、超时或返回空身份时，商家会话不建立，现有会话和数据不被修改。
- 同一成员资格在登录后被停用时，后续受保护操作必须失败，而不是继续沿用旧权限。
- 商家被暂停时，不得签发新会话；已有会话的后续操作也必须失败。
- 商家会话提交另一个商家的记录标识时，系统必须拒绝且不得泄露目标记录内容。
- 多行菜品输入中的空行可忽略；单行字段过长、分类无法识别或重复分隔符必须在预览中逐行报错。
- 批量导入提交期间部分写入失败时，系统必须返回逐行结果；再次提交不得制造同商家重名菜。
- 菜单生成跨越无效日历日或结束日期早于开始日期时必须拒绝。
- 菜单避重目标无法全部满足但每类数量足够时，系统可放宽“近期/同周”偏好，但不得放宽分类数量、启用状态或单餐不重复约束，并应说明放宽项。
- 手动订单的顾客资料、餐次或订单不属于当前商家时必须拒绝。
- 已取消订单的付款、送达、确认和普通编辑操作必须拒绝；再次补同一顾客资料和餐次时可经明确确认恢复为草稿，并清理旧生命周期时间。
- 并发创建同一商家同名菜、同一日期餐次或同一顾客资料餐次订单时，最终只能保留一条符合唯一规则的记录，并向另一请求返回冲突。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供一个实际可运行的 v1 商家入口和业务服务；不得创建不承载本功能的空 workspace，也不得新增第二套内容管理进程。
- **FR-002**: 商家入口必须通过微信身份识别已预置的有效商家成员资格和启用商家；成员停用或商家暂停时不得登录或继续操作；不得自动注册商家、成员或任意微信身份。
- **FR-003**: 单一有效成员资格必须直接选中其商家；多个有效成员资格必须让用户显式选择，签发的会话一次只代表一个商家。
- **FR-004**: v1 商家会话必须与共享内容管理后台会话、旧项目产品会话完全分离，且至少包含当前成员、当前商家、商家角色和明确过期时间。
- **FR-005**: 所有 v1 商家读写必须从已验证会话推导当前商家和成员，不得信任客户端提交的商家归属；会话失效或成员停用后必须拒绝访问。
- **FR-006**: 共享内容管理边界必须只承载 v1 数据持久化转换和归属校验，不得承载菜单或订单状态机决策，也不得复用旧项目业务入口。
- **FR-007**: 系统必须支持按当前商家列出、新增、编辑、停用和恢复菜品，不提供物理删除菜品的能力。
- **FR-008**: 菜品必须包含非空菜名、可选主料、荤/素/汤分类和启用状态；同一商家菜名唯一，不同商家可使用相同菜名。
- **FR-009**: 系统必须支持多行菜品文本解析预览；预览逐行给出解析值、错误和同商家重名冲突，确认提交前不得写入。
- **FR-010**: 重名导入默认跳过；只有用户在预览中明确选择覆盖时才更新现有菜品，且提交结果必须逐行说明创建、覆盖、跳过或失败。
- **FR-011**: 菜单生成只能使用当前商家已启用菜品，单餐固定为 2 荤、2 素、1 汤，不得在同一餐重复菜品，不得虚构菜品。
- **FR-012**: 系统必须支持生成单餐和一段日期范围内所选午/晚餐菜单；同一商家、日期、餐次最终只有一个餐次记录。
- **FR-013**: 生成必须优先避开目标日期前 7 个日历日使用过的同菜或同主料；在菜品池足够时，批量生成还必须避免同周重复同一道菜和同一天重复同一主料；不足以满足偏好时可确定性放宽偏好并告知用户，但不能生成分类不完整的菜单。
- **FR-014**: 菜单必须保存生成时的菜名、主料和分类快照；菜品后续编辑或停用不得改写历史快照。
- **FR-015**: 再次生成已有餐次必须二次确认；换菜只能修改目标菜单项，使用当前商家同分类的启用菜品，并在无候选时保持原菜单。
- **FR-016**: 系统必须支持按当前商家及日期/餐次查看订单明细和汇总；草稿可见但只有已确认且未取消订单计入备餐、未付和未送口径。
- **FR-017**: 商家手动补单必须选择当前商家的餐次，并选择已有顾客资料或创建新的“称呼 + 地址”整体资料；新资料的微信身份允许为空且不得自动猜测绑定。
- **FR-018**: 手动订单必须保存顾客称呼、地址、单价快照、份数、备注和手动来源；单价取餐次价格并在其为空时回退商家默认价格；同一商家、餐次、顾客资料重复补单必须显式转为更新现有订单。
- **FR-019**: 商家必须能够修改未取消订单的份数、称呼/地址快照和备注；修改已确认订单前必须二次确认，资料本身默认不随订单快照修改。
- **FR-020**: 订单业务状态只允许草稿转为已确认、草稿/已确认转为已取消，或已取消订单在再次补同一顾客资料和餐次且用户明确确认后重置为草稿；重置必须清理旧确认、取消、付款和送达时间，且所有状态变化保留同一订单记录。
- **FR-021**: 只有已确认订单可以切换付款或送达状态；付款状态必须支持未付与已付双向切换，送达状态必须支持待送与已送双向切换，两组状态互不影响并同步维护相应时间。
- **FR-022**: 系统必须支持对用户明确选中的当前商家订单批量标已送，不得修改未选中、已取消或其他商家订单。
- **FR-023**: 系统必须生成可查看和复制的纯文本备餐/送餐清单，仅统计 `confirmed` 订单，包含餐次、总份数、顾客、份数和地址，不计入 `draft` 或 `canceled` 订单。
- **FR-024**: 所有跨商家读取、关系引用和状态变更必须拒绝；拒绝响应不得暴露另一商家的记录内容。
- **FR-025**: M1 必须复用 M0 已有七个 v1 业务实体和共享内容管理实例；除非规划阶段证明现有字段无法表达本规格，否则不得新增业务实体、数据库或常驻内容管理进程。
- **FR-026**: 本功能必须先补会失败的契约、权限、领域规则和关键用户流程测试，再实现对应行为，并通过仓库完整质量门禁。

### Key Entities

- **商家成员资格**: 表示某微信身份可以经营某个商家；同一微信身份可以拥有多个成员资格，但一次会话只选择一个商家。
- **菜品**: 当前商家的可选菜单元素，包含菜名、主料、分类和启用状态；停用只影响未来生成。
- **餐次**: 当前商家某个日历日的午餐或晚餐，包含菜单快照、价格和订餐状态；同一日期餐次唯一。
- **菜单项快照**: 餐次生成时保存的菜品引用、菜名、主料和分类，历史不会随菜品编辑而变化。
- **顾客资料**: 当前商家下的一条完整“称呼 + 地址”资料，可选绑定微信身份；M1 只由商家选择或创建。
- **订单**: 某顾客资料在某餐次的份数和经营状态，保存顾客、单价与地址快照，并分别维护业务、付款和送达状态。

### Scope Boundaries

- M1 不创建预订批次、不生成或分享顾客卡片。
- M1 不提供顾客身份、顾客资料自助维护、顾客下单或顾客订单页面。
- M1 不接在线支付、退款、配送路线或地图。
- M1 不解析微信群接龙，不提供 agent/AI 菜单或订单能力。
- M1 不做多商家自助开通，不改变共享内容管理后台的既有管理员身份。
- M1 不物理删除菜品、顾客资料、餐次或订单。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 已预置的桃子可在 30 秒内进入唯一商家空间；未预置身份 100% 无法进入或读取商家数据。
- **SC-002**: 桃子可在 2 分钟内完成从粘贴 50 行菜品到预览并确认提交；每一行都有明确结果且同商家不产生重名记录。
- **SC-003**: 在菜品池足够时，单餐生成在 3 秒内返回恰好 2 荤、2 素、1 汤，单餐重复菜和停用菜数量均为 0。
- **SC-004**: 桃子可一次为连续 5 个工作日的午晚餐生成 10 个餐次；菜品池足够时同周重复菜和同日重复主料数量均为 0。
- **SC-005**: 桃子可在 2 分钟内完成“创建手动订单 → 确认 → 标已付 → 标已送”，且每一步刷新后状态和汇总一致。
- **SC-006**: 自动化验证中的跨商家读取、写入、关系引用和批量操作必须 100% 被拒绝，且响应不包含另一商家的业务字段。
- **SC-007**: 菜品、菜单和订单三个切片可分别独立演示；完成任一切片后，先前切片及旧项目现有入口仍全部通过回归验证。
- **SC-008**: M1 完成时只增加一个 v1 商家前端和一个 v1 业务服务，不增加内容管理进程、数据库或 M2/M3 功能入口。

## Assumptions

- MVP 生产环境仍只服务桃子一个商家，但测试和会话契约覆盖同一微信身份拥有多个商家成员资格的情况。
- 桃子 seller/operator seed 已由 M0 建立；本功能不提供自助注册或成员管理。
- 所有业务日期按 Asia/Shanghai 解释，持久化日期继续使用日历日语义。
- 桃子默认菜单结构固定为 2 荤、2 素、1 汤；本功能不提供结构配置界面。
- “近期”固定指目标日期前 7 个日历日；它与同周、同日主料规则都是可放宽偏好，单餐结构和不重复规则不可放宽。
- 菜品批量录入使用简单分隔符规则；无法可靠解析的行由用户在预览中修正，不引入智能推断。
- M1 使用现有 M0 数据实体即可表达需求；若规划发现必须改变数据契约，必须在同一规格中明确原因、兼容性和验证任务。
- 商家需要稳定网络连接才能登录和保存；网络失败时保留页面输入并允许重试，但不承诺离线编辑。
