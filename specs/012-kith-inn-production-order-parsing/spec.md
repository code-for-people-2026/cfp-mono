# 功能规格：kith-inn 生产接龙解析与订单对账

**功能分支**: `012-kith-inn-production-order-parsing`

**创建日期**: 2026-07-11

**状态**: 草稿

**输入**: GitHub issue [#155](https://github.com/code-for-people-2026/cfp-mono/issues/155) 及产品补充：接龙是桃子发布菜单后从微信群收回的订单快照；重复粘贴以最后一次为准但必须确认，同时允许自然语言单独补单。

## 项目作用域

**项目**: kith-inn（非 kith-inn-v1）

**允许触碰的源码路径**:

- `apps/kith-inn-be/**`

- 菜名级订单明细；一份仍表示桃子当前唯一的套餐商品
- 顾客别名自动合并、通用导入框架或面向未来商家的解析抽象
- 新客缺地址时的最终写入守卫，由 #156 负责；本功能沿用现有新客补地址流程

## Clarifications

### Session 2026-07-11

- 接龙由桃子从已发布菜单复制到微信群，模板头的日期、周几和餐次用于定位订单；菜单正文只供顾客查看，回贴解析时必须忽略，且不要求与当前系统菜单一致。
- 再次粘贴同一天的完整接龙是全量快照，以最后一次为准；系统必须先展示差异并由桃子确认，不能直接覆盖。
- 一句自然语言补单是增量修改，只触碰明确指定的顾客、日期和餐次，不能清除同日其他订单。
- Q: 后续完整接龙快照是否也覆盖同一日期餐次下、由自然语言补入但未出现在群接龙中的订单？ → A: 覆盖；完整接龙是目标范围内所有订单的最终快照，确认卡必须明确列出将退出有效口径的自然语言补单。
- Q: 自然语言“加王阿姨 2 份”命中已有同坐标订单时，2 份表示最终总数还是在现有数量上再增加 2 份？ → A: “加 N 份”结合当前订单按追加理解：无现单时从 0 增加为 N，已有现单时增加 N；“改成/总共 N 份”才设置最终总数。无论哪种解释，都必须展示当前值、动作和结果供桃子确认。

## 用户场景与测试

### 用户故事 1 - 从完整接龙得到可信订单候选（优先级：P1）

桃子把微信群里的完整接龙粘回「今天」对话后，看到按日期、餐次、顾客和份数拆好的订单候选。模板菜单可以已经与系统当前菜单不同，但不影响统计；日期、周几或餐次不完整、有冲突时，系统要求她补充或检查，不能给出可执行写入确认。

**优先级理由**: 接龙是主要收单入口；日期或午晚餐错配会直接造成漏做、错送。

**独立测试**: 粘贴至少十段真实接龙，比较每条候选的日期、餐次、顾客和份数，并对日期缺失、周几冲突、餐次歧义和菜单噪声分别验证失败关闭或正确忽略。

**验收场景**:

1. **假设** 接龙模板含明确日期、匹配的周几、午晚餐标题、编号菜单和顾客留言，**当** 桃子整段粘贴，**则** 系统只把顾客留言解析成订单候选，并按留言餐次映射到对应标题日期。
2. **假设** 一个顾客留言同时订午餐和晚餐，**当** 两个餐次都能由标题定位日期，**则** 系统拆成两个独立候选。
3. **假设** 菜单正文与系统当前菜单不同，**当** 桃子粘贴接龙，**则** 解析结果不因菜名、菜单顺序或菜品池变化而改变。
4. **假设** 日期缺失、日期非法、日期与周几冲突，或多餐模板中的留言无法唯一确定餐次，**当** 系统解析，**则** 返回具体补全或纠错提示，不生成可执行写入确认。
5. **假设** 模板含“例 桃子 1份午餐晚餐”和编号菜名，**当** 系统解析，**则** 示例和菜名都不被识别为真实订单。

---

### 用户故事 2 - 用最新完整接龙对账（优先级：P1）

桃子已经粘贴过一次接龙，顾客继续追加、改数量或撤回后，她再次粘贴相同日期范围的完整接龙。系统把这次内容视为该范围的最新快照，展示将新增、更新、取消和保持不变的订单；只有桃子确认后才一次性应用。

**优先级理由**: 群接龙会持续变化；只新增会制造重复或保留过期份数，直接扭曲备餐、送餐和收款口径。

**独立测试**: 先导入一份接龙，再导入包含新增、改量、撤回和未变化顾客的新版接龙；确认前数据不变，确认后目标范围与新快照一致，其他日期餐次不受影响。

**验收场景**:

1. **假设** 同一顾客、日期和餐次已有 active 订单，**当** 新接龙里的份数不同，**则** 确认卡标明旧值与新值，确认后更新原订单而不新增重复订单。
2. **假设** 新接龙增加了顾客，**当** 桃子确认，**则** 新增对应订单；新客仍遵循既有身份和地址确认流程。
3. **假设** 任意来源的 active 订单不再出现在新快照的相同日期餐次范围，包括此前自然语言补单，**当** 桃子确认，**则** 系统按已确认的快照覆盖语义让该订单退出当前有效口径，而不是物理删除历史。
4. **假设** 差异涉及已确认订单，**当** 展示确认卡，**则** 明确提示会同步影响订单台账、送餐清单和收款汇总；桃子未确认前所有数据保持不变。
5. **假设** 预览后、确认前目标订单已被其他操作改变，**当** 桃子点击旧确认卡，**则** 系统拒绝套用过期差异并要求重新预览。
6. **假设** 一次对账包含多项变化且任一写入失败，**当** 操作结束，**则** 整次快照不留下部分更新。

### 用户故事 3 - 用自然语言单独补单（优先级：P1）

桃子在接龙导入之后收到私聊单，可以说“7 月 13 日晚餐，加王阿姨 2 份”。系统只预览这一笔增量变更，信息完整且桃子确认后才写入，不会把同一天接龙里的其他顾客当作过期数据清除。

**优先级理由**: 私聊补单是已知高频漏单来源，必须与完整接龙的全量语义明确分开。

**独立测试**: 先导入多人接龙，再用自然语言新增或修改一个明确坐标；确认后只有目标订单变化，同日其他订单保持不变。

**验收场景**:

1. **假设** 桃子提供明确动作、日期、餐次、顾客和份数，**当** 系统识别为单独补单，**则** 确认卡只展示该业务坐标的变化。
2. **假设** 自然语言缺少日期、餐次、顾客、份数或明确动作，**当** 系统解析，**则** 返回补全提示，不生成可执行写入确认。
3. **假设** 目标坐标不存在 active 订单，**当** 桃子确认，**则** 新增一张草稿订单。
4. **假设** 目标坐标已有 1 份 active 订单，**当** 桃子说“加 2 份”，**则** 确认卡展示“当前 1 份 + 2 份 → 共 3 份”，确认后更新为 3 份且不改变同日其他订单。
5. **假设** 目标坐标已有 active 订单，**当** 桃子说“改成 2 份”或“总共 2 份”，**则** 确认卡展示当前份数与目标总数，确认后设置为 2 份。
6. **假设** 目标订单已确认，**当** 系统展示更新预览，**则** 明确提示对备餐、履约和收款口径的影响，未经确认不修改。

### 用户故事 4 - 生产与评测使用同一解析口径（优先级：P1）

维护者可以用桃子的真实接龙样本衡量生产实际解析能力，不再出现“评测通过的解析器没有接入生产”或评测漏掉日期的情况。

**优先级理由**: 解析结果依赖模型；只有生产和评测同路，准确率指标才有上线意义。

**独立测试**: 对生产使用的解析入口运行真实模型评测，按四字段整条匹配统计，并记录样本数、准确率、午晚错配和运行时间。

**验收场景**:

1. **假设** 至少十段已人工标注日期、餐次、顾客和份数的真实接龙，**当** 运行评测，**则** 每条只有四字段全对才计为正确。
2. **假设** 生产解析行为发生变化，**当** 运行相同评测，**则** 结果直接反映该变化，不经过另一套平行解析逻辑。
3. **假设** 模型输出缺字段、非法值或无法通过严格校验，**当** 生产或评测调用解析器，**则** 都按失败关闭处理。

### 边界情况

- 同一完整接龙可包含午餐和晚餐，也可让两个餐次对应不同日期；每条订单按餐次映射到相应标题日期。
- 年份省略时，以 Asia/Shanghai 的操作参考日解析为明确年份，并在确认卡展示完整日期；无法唯一确定时要求补充。
- 只有一个餐次标题且留言未写餐次时可继承该标题；存在多个可能餐次时不得猜测。
- 周几是日期一致性校验信息，不单独存入订单；明确提供却与日历不符时整次快照不可执行。
- 没有真实顾客订单的空接龙可能意味着清空目标范围，也可能是未完成粘贴；在没有明确、可确认的清空预览前不得执行。
- 解析器无法确认某行是菜单还是顾客订单，或发现疑似订单行无法完整解析时，不得用可能不完整的快照触发批量取消。
- 同名顾客继续沿用现有名字归一与人工确认语义，本功能不自动推断别名关系。
- 已取消历史订单不占 active 业务坐标；后来重新下单可创建新草稿。
- 多次快速提交相同快照、重复点击确认或网络重试，不得产生重复订单或重复履约记录。

## 需求

### 功能需求

- **FR-001**: 系统必须区分“完整接龙快照”和“自然语言单笔增量”，并在确认卡中明确显示本次操作模式。
- **FR-002**: 完整接龙解析必须输出本次快照覆盖的日期与餐次范围，以及每条候选的顾客、日期、餐次和份数。
- **FR-003**: 系统必须从模板标题取得日期、周几和餐次上下文；顾客留言中的明确餐次优先用于匹配相应标题；菜单正文、价格、示例行和空白编号必须忽略。
- **FR-004**: 系统不得使用接龙菜单正文恢复或校验系统菜单，也不得把菜名转换成订单明细。
- **FR-005**: 年份省略的日期必须按 Asia/Shanghai 的操作参考日解析成明确日期并展示；无效日期、日期与周几冲突或无法唯一映射时必须失败关闭。
- **FR-006**: 任何准备写入的候选必须同时具有有效的顾客、日期、餐次和正整数份数；缺失、非法或有歧义时不得生成可执行确认。
- **FR-007**: 生产聊天必须复用被真实样本评测的唯一接龙解析行为，不得保留一条只供评测、与生产结果无关的平行解析链路。
- **FR-008**: 完整接龙预览必须读取目标范围全部当前 active 订单且不区分此前录入方式，并把差异明确分为新增、更新、退出有效口径和未变化；更新项必须展示旧值与新值。
- **FR-009**: 完整接龙的写入必须以桃子确认的最新快照为准，在同一原子操作中应用全部差异；确认前不得写订单、明细、餐次或履约状态。
- **FR-010**: 同一卖家、顾客、日期和餐次命中 active 订单时必须更新现单，不得创建第二张 active 订单；已取消历史单不阻止重新下单。
- **FR-011**: 快照中退出有效口径的订单必须保留审计历史；若原订单已确认，其履约和经营汇总必须同步退出。
- **FR-012**: 含已确认订单变化的预览必须突出说明对订单台账、送餐清单和收款汇总的影响，并要求桃子明确确认。
- **FR-013**: 确认操作必须绑定预览时的输入和数据版本；目标数据在预览后变化时，旧确认不得覆盖新状态。
- **FR-014**: 自然语言补单必须要求明确的动作、日期、餐次、顾客和份数，并只新增或更新唯一定位的业务坐标；“加 N 份”表示在当前数量上增加 N，“改成/总共 N 份”表示把最终数量设置为 N。
- **FR-015**: 自然语言补单不得把未提及的同日订单视为删除候选，也不得改变其他日期、餐次或顾客的订单。
- **FR-016**: 自然语言补单确认卡必须展示系统理解的当前数量、增加量或目标总数以及确认后的最终数量；不得仅复述含糊原句让桃子自行推断。
- **FR-017**: 完整接龙或自然语言补单涉及新顾客时，必须沿用现有的新客确认流程，并保持本次解析出的日期、餐次和份数不变。
- **FR-018**: 疑似订单行未完整解析、快照范围不明确或空快照意图不明确时，系统必须要求桃子检查，禁止据此批量取消现有订单。
- **FR-019**: 系统必须对重复确认、网络重试和并发提交保持幂等，并保证快照对账失败时不留下部分更新。
- **FR-020**: 评测必须使用至少十段桃子真实接龙，按日期、餐次、顾客和份数四字段整条匹配，并单独统计午晚餐错配。
- **FR-021**: 本功能必须沿用现有卖家隔离、订单生命周期、套餐商品和确认卡机制，不新增平行状态机或通用导入系统。

### 关键实体

- **接龙快照**: 桃子从微信群完整复制回来的某组日期与餐次最终订单清单；包含原文、覆盖范围、解析候选和无法安全解析的片段，并覆盖该范围内此前由接龙或自然语言产生的 active 订单。
- **单笔增量命令**: 桃子用自然语言明确指定的一次新增或数量修改；只作用于一个顾客、日期和餐次坐标，并区分相对增加量与绝对目标总数。
- **订单候选**: 尚未写库的顾客、日期、餐次、份数四元组，并带有输入模式和解析状态。
- **对账差异**: 当前 active 订单与新快照之间的新增、更新、退出有效口径和未变化集合；只有确认后才能应用。
- **业务坐标**: 卖家、顾客、日期、餐次的组合，用于识别唯一 active 订单。
- **订单**: 一个顾客在一个日期和餐次的最终有效份数及生命周期记录；取消后保留历史但退出 active 口径。

## 成功标准

### 可衡量结果

- **SC-001**: 在至少十段真实接龙上，日期、餐次、顾客和份数四字段整条匹配准确率达到 95% 以上，午餐与晚餐错配为 0。
- **SC-002**: 对缺日期、非法日期、周几冲突、餐次歧义、疑似订单漏解析和模型非法输出的自动化样本，100% 不产生可执行写入确认。
- **SC-003**: 对包含新增、更新、退出和未变化项的重复快照，在确认前数据库变化为 0；确认成功后目标范围与已确认快照一致且 active 重复订单数为 0。
- **SC-004**: 对快照写入的每个失败点进行故障注入后，100% 保持操作前状态，不出现部分更新、孤立明细或订单与履约状态不一致。
- **SC-005**: 在已有同日多人订单的情况下执行至少十种自然语言单笔补单，100% 只改变明确目标坐标，且确认卡展示的计算结果与确认后的最终份数一致，其他订单保持不变。
- **SC-006**: 同一确认请求因重复点击或网络重试提交两次、或两次同时到达时，最终只应用一次；目标数据已经被其他操作改变时，旧确认必须被拒绝并要求重新预览。
- **SC-007**: 真实模型评测报告记录模型、参考日期、样本数、四字段准确率、午晚错配和运行时间，并直接覆盖生产解析入口。
- **SC-008**: 所有现有菜单发布、订单确认、取消、送餐和收款成功路径继续通过仓库质量门禁，且不要求桃子增加确认卡以外的手工清理步骤。

## 假设

- 桃子发布出的模板通常包含日期、周几、餐次和菜单正文；顾客只在后续编号中追加姓名、餐次和份数。
- 完整接龙是覆盖范围内全部订单的最终全量清单，不区分此前通过接龙还是自然语言录入；自然语言补单是单坐标增量。自然语言补单若未出现在后续完整接龙中，会作为退出项展示并在桃子确认后退出有效口径。
- 桃子确认差异卡即构成写入授权；涉及已确认订单时，同一卡片必须额外突出经营影响。
- 自然语言中的“加”按相对增量处理，“改成/总共”按绝对总数处理；模型只负责理解表达，最终计算结果必须在写入前由桃子确认。
- 现有一份对应一个套餐商品的业务模型保持不变，菜单菜名不是订单明细来源。
- 项目尚未正式部署；不需要为历史开发数据提供通用迁移或修复工具。
- 订单原子性基础由 #154 提供，本功能在其上增加跨多张订单的快照对账边界。
