# Feature Specification: kith-inn 主链路真实 E2E 与 CMS 集成验证

**Feature Branch**: `016-kith-inn-mainline-e2e`

**Created**: 2026-07-13

**Status**: Draft

**Input**: GitHub issue #157：仅针对 kith-inn（非 kith-inn-v1），建立 H5→BE→CMS→PostgreSQL 的真实主链路验证闭环。

## 项目作用域

本功能验证现有 kith-inn MVP 是否能从桃子的 H5 操作贯通到真实业务数据，而不是新增业务能力。目标是让 PR 门禁能够证明以下已实现能力在生产等价数据链路上协同工作：登录、粘贴完整接龙、确认预览与草稿、确认订单、菜单生成/换菜/发布、标已付和批量送达。

本功能只覆盖旧 `kith-inn`。不得修改、借用或重置 `kith-inn-v1` 的业务数据与测试用例；不得用 mock CMS 请求替代要求中的真实 CMS internal route 与 PostgreSQL 证据。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 从接龙形成可经营订单 (Priority: P1)

作为桃子，我希望登录后粘贴完整接龙，先看清系统理解，再确认草稿和订单，使订单台账与送餐任务准确出现而没有重复或半成品。

**Why this priority**: 记单和确认订单是后续菜单、收款与送餐的业务起点；如果只测单层 mock，无法证明 H5、BE、CMS 与数据库的契约仍然一致。

**Independent Test**: 从独立初始数据登录 H5，粘贴一段含午晚餐和多名顾客的固定接龙，确认解析预览、草稿和订单；直接读取最终经营结果即可验证订单坐标、数量、状态和履约任务。

**Acceptance Scenarios**:

1. **Given** 桃子已登录且测试卖家只有独立 seed 数据，**When** 她粘贴包含明确日期、午晚餐、顾客与份数的完整接龙并确认，**Then** 系统先展示无写入的差异预览，再为每个顾客、日期和餐次形成唯一草稿。
2. **Given** 草稿已形成，**When** 桃子确认订单，**Then** 对应餐次进入经营口径、订单变为已确认且每张订单恰有一条待送任务。
3. **Given** 同一完整接龙再次提交，**When** 桃子重复确认或重试确认订单，**Then** 最终业务结果与首次成功相同，不新增 active 重复订单、重复明细或重复履约。

---

### User Story 2 - 从菜单走到当餐收尾 (Priority: P1)

作为桃子，我希望为已有经营订单生成菜单、换掉一道菜、发布菜单，再把订单标为已付并批量送达，从而确认一餐可以完整收尾。

**Why this priority**: 这是 MVP 从订单到执行闭环的另一半，覆盖菜单、收款和送餐三个确定性详情入口的共享后端操作。

**Independent Test**: 以独立 seed 加少量已确认订单为起点，通过 H5 完成菜单生成、单位置换菜、发布、收款和批量送达；最终核对菜单状态、菜品差异、付款状态和履约状态。

**Acceptance Scenarios**:

1. **Given** 目标餐次已有已确认订单和足够菜品，**When** 桃子生成菜单，**Then** 只使用她启用的菜品形成一份目标餐次草稿菜单。
2. **Given** 菜品池较小且仍有合法候选，**When** 桃子自动换掉目标位置，**Then** 只改变该位置并显示实际放宽的中文原因。
3. **Given** 菜单确认无误，**When** 桃子发布菜单、把目标订单标为已付并批量标记送达，**Then** 菜单进入已发布状态，目标订单退出未付口径，目标履约全部进入已送达状态。

---

### User Story 3 - 错误输入与重试保持安全 (Priority: P1)

作为桃子，我希望缺少关键日期时系统拒绝执行，而顾客缺地址时仍可继续；网络重试或重复点击也不能制造重复经营数据。

**Why this priority**: 缺日期会误记到错误餐次，缺地址却是允许的真实经营状态；两者必须被自动化清楚区分，重试语义也必须与原子性规格一致。

**Independent Test**: 分别提交缺日期接龙、无地址新客订单、重复接龙确认和重复订单确认，检查页面反馈与数据库前后差异。

**Acceptance Scenarios**:

1. **Given** 输入无法得到唯一日期，**When** 桃子请求记单，**Then** 系统明确要求补充日期且业务数据变化为零。
2. **Given** 新顾客没有地址，**When** 桃子确认其订单，**Then** 订单和履约正常形成并出现在“无地址”兜底组，不增加补地址阻断。
3. **Given** 客户端重复提交同一确认或首次结果未知后重试，**When** 请求一次或并发多次到达，**Then** 最终只应用一次且返回可识别的已完成结果。

---

### User Story 4 - 门禁证明真实租户边界 (Priority: P1)

作为仓库维护者，我希望相关 PR 自动运行真实 CMS 与 PostgreSQL 集成证据，并证明一个卖家无法观察或修改另一个卖家的数据，失败时可以直接取得诊断材料。

**Why this priority**: 租户隔离和真实数据链路是单元 mock 无法替代的安全门禁；测试若未被 CI 实际发现，也不能形成交付保障。

**Independent Test**: 在同一独立数据库准备两个卖家及各自 operator、顾客、订单、菜单与履约，通过真实 internal routes 分别读写并尝试跨卖家引用；核对返回结果和数据库不变量，再验证 PR 门禁确实执行目标 suite。

**Acceptance Scenarios**:

1. **Given** 卖家 A 与卖家 B 都有同类业务数据，**When** A 登录后查询订单、菜单、顾客或送餐数据，**Then** 结果只包含 A 的数据。
2. **Given** A 知道 B 的资源标识，**When** A 尝试读取、更新或建立跨卖家 relationship，**Then** 请求失败关闭且 B 的数据变化为零。
3. **Given** PR 修改 kith-inn H5、BE、Payload 模块或相关 CMS 路径，**When** CI 计算受影响测试，**Then** 本 suite 被执行；失败时保留可定位到场景和步骤的 trace/report。

### Edge Cases

- 测试开始前存在上一次失败遗留的数据、进程或报告时，新的运行必须重建目标环境，不能误用旧成功结果。
- 接龙日期跨 Asia/Shanghai 的日界线或与周几冲突时，必须沿用生产日期校验，不能由测试时钟默默修正。
- 同一顾客同时有午餐和晚餐时，两餐必须保持为不同订单与不同履约，不得在断言中合并数量。
- 菜品池缩小到仅一个替换候选时仍应成功；候选为零时才可返回无替代菜。
- 批量送达中混入另一卖家、已取消或已送达项时，不得误改非目标项。
- CI 同时选择多个 CMS-backed E2E suite 时，不得因端口、数据库或共享构建目录竞争产生随机失败。
- 本地开发者未预先启动数据库时，标准验证入口必须给出可执行准备方式或自动完成准备，不能静默降级为 SQLite/mock 证据。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 为 `@cfp/kith-inn-fe` 保留可由根级 E2E 命令发现的 H5 自动化入口，并把新增主链路场景纳入同一受支持入口。
- **FR-002**: 主链路 suite MUST 使用独立、可重复重建的 kith-inn 测试数据，并自动编排所需 H5、BE、CMS 与 PostgreSQL；不得依赖开发者当前登录态或已有业务数据。
- **FR-003**: suite MUST 通过真实 H5 操作覆盖登录、粘贴接龙、预览确认、草稿形成与订单确认，并验证最终订单、明细、餐次和履约状态。
- **FR-004**: suite MUST 覆盖菜单生成、自动换菜、发布、标已付和批量送达，并验证换菜只改变目标位置、放宽原因可见及最终经营状态一致。
- **FR-005**: 缺少或无法唯一确定日期的记单输入 MUST 失败关闭，展示可理解的补全提示，且订单、明细、餐次和履约变化数均为零。
- **FR-006**: 缺少地址的顾客和订单 MUST 能完成既有记单、确认和履约流程，并进入明确的无地址兜底展示。
- **FR-007**: 重复接龙提交、重复确认点击、结果未知后的重试与并发确认 MUST 保持幂等，不得产生第二张 active 订单、第二套明细或第二条有效履约。
- **FR-008**: 至少一组自动化证据 MUST 不 mock CMS fetch，必须贯通真实 CMS internal routes 并把业务结果持久化到 PostgreSQL。
- **FR-009**: 真实集成证据 MUST 准备至少两个卖家，并验证读取、更新、批量操作和 relationship 写入的租户隔离；跨卖家观察或数据变化数必须为零。
- **FR-010**: 每个场景 MUST 从可识别的初始状态开始并可单独重跑；一次失败不得污染后续场景或另一项目数据。
- **FR-011**: CI MUST 在 kith-inn H5、BE、Payload package 或相关 CMS 配置、seed、internal route 与共享 helper 改变时执行目标 suite，同时避免无关项目 PR 被迫运行 kith-inn E2E。
- **FR-012**: 多个 CMS-backed E2E 同时受影响时 MUST 隔离端口、数据库与共享构建状态，或以确定性顺序运行，禁止依赖竞态恰好不发生。
- **FR-013**: CI 失败 MUST 上传 Playwright trace、测试报告和失败步骤所需的服务日志；成功运行不得遗留会被后续运行误读的报告。
- **FR-014**: 测试中的接龙解析 MUST 使用生产解析入口与生产校验语义；允许在外部模型边界使用固定响应保证确定性，但不得建立只供测试的平行业务解析器。
- **FR-015**: 本功能 MUST 沿用现有订单、菜单、收款、履约、原子性、地址选填和自适应换菜契约，不新增业务 schema、状态机或面向未来的功能。
- **FR-016**: 本功能 MUST NOT 修改、导入、复用或重置 `kith-inn-v1` 业务用例与数据；对 v1 哨兵数据的内容、数量和标识变化必须为零。
- **FR-017**: 实现 MUST 以可独立 review 和验证的 PR 切片交付；规格、真实 CMS/PostgreSQL 基座、订单链路、菜单/收尾链路和最终失败/租户门禁不得无边界合并为单个大 PR。

### Key Entities

- **E2E Scenario**: 一个可独立运行的用户旅程，包含初始数据、H5 操作、跨层检查点、最终不变量与失败证据。
- **Test Seller / Operator**: 测试登录与租户信任根；至少两组并存以证明数据隔离。
- **Order Aggregate**: 顾客、订单、订单明细、餐次与履约组成的经营一致性结果。
- **Menu Aggregate**: 目标餐次、菜单计划、菜品位置、发布状态和放宽原因组成的菜单结果。
- **Run Evidence**: 场景结果、trace、report 与必要服务日志，绑定到单次运行且不可被旧产物冒充。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 主成功链路从登录到批量送达的全部规定步骤在独立环境中连续运行 3 次，通过率为 100%。
- **SC-002**: 成功链路结束后，每个卖家、顾客、日期、餐次的 active 订单数恰为 1，每张已确认订单的有效履约数恰为 1，非目标菜单位置变化数为 0。
- **SC-003**: 缺日期、日期冲突和不可唯一定位输入的自动化样本中，可执行写入确认产生数为 0，业务数据变化数为 0。
- **SC-004**: 无地址新客从记单到确认和送达的完成率为 100%，强制补地址阻断数为 0，且无地址兜底展示覆盖率为 100%。
- **SC-005**: 对同一接龙和确认操作执行重复、重试与并发提交后，重复 active 订单、重复明细集合和重复有效履约数均为 0。
- **SC-006**: 双卖家真实集成场景中，跨租户读取结果数、跨租户成功写入数及另一卖家数据变化数均为 0。
- **SC-007**: 对任一相关路径的代表性 PR diff，CI 选择目标 suite 的覆盖率为 100%；对已声明无关的代表性项目 diff，误选 kith-inn E2E 的次数为 0。
- **SC-008**: 任一 E2E 断言失败后，CI 产物中可取得对应场景的 trace/report 与失败阶段，诊断产物缺失率为 0。
- **SC-009**: 所有新增验证与全仓 `pnpm verify` 均通过，`kith-inn-v1` 文件和测试数据的业务变化数为 0。

## Assumptions

- #154、#155、#156 与 #163 已完成，原子订单生命周期、生产接龙解析、地址选填和自适应换菜是本功能要验证的既有契约。
- CI 已提供 PostgreSQL 服务；本地验证可以复用仓库标准容器入口，但不能把“本地未启动数据库”静默降级成 SQLite 通过。
- 微信生产 code 换 openid 不在自动化范围；H5 使用现有开发登录入口建立与生产等价的 operator→seller 会话，之后所有业务请求走真实鉴权和租户解析。
- 外部 LLM 不作为门禁可用性依赖；固定模型响应只替代外部不确定边界，生产 parser、校验、预览、确认和持久化路径保持不变。
- 菜单群通知只验证现有发布文案与复制前状态，不接入真实微信群；收款只验证手工状态，不接支付。
- 小程序真机与生产部署验收属于 #158，不纳入本功能。
