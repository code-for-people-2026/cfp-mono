# Feature Specification: kith-inn 配送地址选填与自动带出

**Feature Branch**: `codex/docs-kith-inn-optional-address`

**Created**: 2026-07-13

**Status**: Corrected requirements; ready for planning

**Input**: User description: "配送地址并不是必填项。桃子自己记得顾客地址，顾客接龙时不会每次贴地址；桃子可以选填，填过后同一个人的后续订单自动带地址，没填也可以正常记单和确认。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 地址留空也能完成经营操作 (Priority: P1)

作为桃子，我可以在新顾客或订单没有配送地址时照常记单和确认，不需要为了系统重复输入自己已经记得的信息。

**Why this priority**: 地址不是接龙的有效必填信息；把它作为确认前置条件会让桃子的真实操作变慢，并错误阻断正常订单。

**Independent Test**: 为没有默认地址的顾客创建订单并留空地址，随后确认；订单正常进入 confirmed，相关餐次和送餐任务正常产生，送餐展示使用无地址兜底。

**Acceptance Scenarios**:

1. **Given** 新顾客地址输入为空，**When** 桃子确认接龙变更，**Then** 系统创建没有默认地址的顾客和没有地址快照的订单，不要求补录。
2. **Given** 一张没有地址快照的 draft order，**When** 桃子从订单页或对话确认，**Then** 系统按既有规则确认并生成送餐任务，不返回缺地址错误。
3. **Given** 送餐任务对应的订单没有地址，**When** 桃子查看送餐清单，**Then** 系统以明确的无地址兜底展示该任务，不丢单也不阻断操作。

---

### User Story 2 - 保存一次后续订单自动带地址 (Priority: P1)

作为桃子，我可以在方便时为顾客选填一个默认地址；以后同一个人再次下单，即使接龙里没有地址，新订单也自动带出这个地址。

**Why this priority**: 一次记录、后续复用可以减少重复输入，同时仍尊重接龙本身通常不含地址的事实。

**Independent Test**: 首次为新顾客填写地址并创建订单；下一次仅用相同顾客名字和订单信息记单，不再提交地址，检查新订单仍带有已保存地址快照。

**Acceptance Scenarios**:

1. **Given** 新顾客没有资料，**When** 桃子在确认卡选填有效地址并确认，**Then** 系统把地址保存为顾客默认地址，并复制到本次订单地址快照。
2. **Given** 顾客已有默认地址，**When** 后续接龙按同一顾客匹配且输入中没有地址，**Then** 新订单自动复制该默认地址为本单快照。
3. **Given** 顾客始终没有默认地址，**When** 该顾客后续再次下单，**Then** 新订单仍可保持无地址并正常处理，系统不反复要求补录。

---

### User Story 3 - 订单地址保持历史快照 (Priority: P2)

作为桃子，我希望每张订单保留创建当时带出的地址；以后顾客默认地址发生变化时，不应悄悄改掉旧订单记录。

**Why this priority**: 订单地址用于还原当次送餐事实；默认地址只服务未来订单，不能覆盖历史。

**Independent Test**: 创建一张带地址订单后修改顾客默认地址，再创建下一张订单；旧订单保持原地址，新订单使用新默认地址。

**Acceptance Scenarios**:

1. **Given** 顾客已有一张带地址的历史订单，**When** 顾客默认地址以后改变，**Then** 历史订单地址快照保持原值。
2. **Given** 同一顾客默认地址已改变，**When** 创建后续订单，**Then** 只有后续订单复制新默认地址。

### Edge Cases

- 地址为 `null`、空字符串或只含空白字符时，都按“未填写”处理，不保存空白默认地址。
- 顾客名字必须继续按既有归一规则匹配；不能因大小写、前后空格或内部多余空格重复建人。
- 同一个新顾客在一次接龙中有多条订单时，只要其中一次填写地址，该顾客本批次的新订单都使用同一地址。
- 已创建订单的地址是快照；完整接龙覆盖数量或自然语言补单不得顺手改写既有快照。
- 无地址送餐任务必须保留在送餐清单中，不能被过滤掉。
- 本功能不得读取或修改 `kith-inn-v1` 数据。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 配送地址必须保持选填，新顾客和订单都不得因地址缺失而无法保存。
- **FR-002**: 系统必须把 `null`、空字符串和纯空白地址统一视为未填写。
- **FR-003**: 缺地址 draft order 必须能够遵循既有确认流程进入 confirmed，并正常创建餐次和送餐任务。
- **FR-004**: 桃子为新顾客填写地址时，系统必须把同一值保存为顾客默认地址和本次新订单地址快照。
- **FR-005**: 创建匹配到既有顾客的新订单时，系统必须把该顾客当时的默认地址复制为新订单地址快照；输入无需重复携带地址。
- **FR-006**: 没有默认地址的既有顾客必须仍可继续创建无地址的新订单，系统不得增加强制补录步骤。
- **FR-007**: 顾客默认地址的后续变化不得追溯改写任何既有订单地址快照。
- **FR-008**: 无地址订单产生的送餐任务必须保留在经营清单中，并使用明确的无地址兜底展示。
- **FR-009**: 订单确认入口不得新增 `missing-address` 错误、补地址前置条件或其他等价阻断。
- **FR-010**: 本功能只能影响 kith-inn 的地址行为与文档，不得读取或修改 `kith-inn-v1` 业务数据。

### Key Entities

- **顾客**: 以接龙名字识别的长期资料；可选保存一个默认地址，供未来新订单带出。
- **订单**: 一个顾客在一个日期和餐次的订单；可选保存创建当时的地址快照，快照不随顾客资料追溯变化。
- **送餐任务**: 订单确认后产生的经营记录；地址来自订单快照，缺失时仍存在并显示无地址兜底。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 对未填写、空字符串和纯空白地址的新顾客订单，记单与确认成功率为 100%，且不会出现补地址阻断。
- **SC-002**: 顾客保存默认地址后，其后续新订单在未重复输入地址时自动带出正确地址的覆盖率为 100%。
- **SC-003**: 顾客默认地址改变后，既有订单地址快照变化数为 0，后续新订单使用新值的正确率为 100%。
- **SC-004**: 无地址 confirmed order 的送餐任务保留率为 100%，并全部出现在无地址兜底分组。
- **SC-005**: 更正后的实现和文档不产生任何 `kith-inn-v1` 文件或数据变化。

## Assumptions

- 桃子在系统外也记得顾客地址；系统地址用于便利和分拣，不是订单合法性的必要条件。
- 顾客不会在每次接龙中重复提供地址，接龙有效信息仍以日期、周几、餐次、顾客和份数为主。
- 地址继续使用自由文本，不拆分楼栋、单元或房号。
- 本功能不新增修改既有顾客地址的交互；若桃子通过既有资料入口修改默认地址，快照规则仍按本规格执行。
- 现有确认生命周期、付款和送达规则保持不变。
