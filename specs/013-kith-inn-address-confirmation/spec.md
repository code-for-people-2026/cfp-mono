# Feature Specification: kith-inn 缺地址确认守卫

**Feature Branch**: `codex/fix-kith-inn-missing-address-confirm`

**Created**: 2026-07-13

**Status**: Ready for implementation

**Input**: User description: "仅针对 kith-inn：缺地址草稿可以保存，但补齐地址前不得确认进入经营口径；确认卡和订单页都要展示缺地址状态，补齐后顾客默认地址与订单地址快照语义一致。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 缺地址订单不能进入经营口径 (Priority: P1)

作为桃子，我可以先保存一个还没拿到地址的草稿，但无论从订单页点击确认还是在对话里口头确认，系统都必须阻止这张订单进入备餐和送餐口径，并告诉我需要先补地址。

**Why this priority**: 这是防止无地址送餐任务进入经营数据的最后安全边界；任何入口绕过都会让送餐清单失去防漏、防错价值。

**Independent Test**: 准备一张地址为空或只有空白字符的草稿，分别从所有确认入口尝试确认；每次都收到同一类可操作提示，订单仍为草稿，相关餐次和送餐任务均未产生或改变。

**Acceptance Scenarios**:

1. **Given** 一张缺地址草稿，**When** 桃子点击确认订单，**Then** 系统拒绝确认，明确提示先补地址，且订单状态、餐次状态和送餐任务保持原样。
2. **Given** 一张缺地址草稿，**When** 桃子通过对话口头确认，**Then** 系统执行与点击确认相同的拒绝规则和提示，不产生任何经营副作用。
3. **Given** 一张地址只包含空格或换行的草稿，**When** 桃子确认，**Then** 系统把它视为缺地址并拒绝。
4. **Given** 一张有非空地址快照且满足其他既有条件的草稿，**When** 桃子确认，**Then** 订单按既有流程进入经营口径，不受本功能额外阻挡。

---

### User Story 2 - 从待补状态快速补齐地址 (Priority: P1)

作为桃子，我看到订单缺地址后，可以直接进入地址补全，保存一次就同时补齐这张订单和该顾客以后下单所用的默认地址，然后继续确认订单。

**Why this priority**: 只阻止确认而不给短路径会把安全规则变成经营阻力；补全必须既修复当前订单，也减少下一次重复录入。

**Independent Test**: 从一张缺地址草稿的待补标识进入补全，填写有效地址并保存；检查当前订单地址快照和顾客默认地址相同，再确认订单并看到它正常进入送餐口径。

**Acceptance Scenarios**:

1. **Given** 缺地址草稿及其顾客也没有默认地址，**When** 桃子填写并保存有效地址，**Then** 当前订单地址快照和顾客默认地址同时更新为该值。
2. **Given** 地址补全成功，**When** 桃子随后确认订单，**Then** 订单可按既有规则确认，并以该订单地址快照进入送餐清单。
3. **Given** 顾客还有其他既有订单，**When** 桃子补齐当前缺地址订单，**Then** 其他订单的地址快照不被改写。
4. **Given** 桃子输入空白地址，**When** 保存，**Then** 系统拒绝保存并保留待补状态。

---

### User Story 3 - 所有相关界面明确展示待补状态 (Priority: P2)

作为桃子，我在订单录入确认卡和订单页都能一眼看出哪张订单缺地址，并能从提示直接开始补全，而不是等到送餐分组里看到“无地址”才发现问题。

**Why this priority**: 越早暴露缺口，桃子越容易在确认前修复；同时保留最终确认守卫作为不可绕过的兜底。

**Independent Test**: 分别查看包含缺地址候选的订单录入确认卡和包含缺地址草稿的订单页；两处都有明确“待补地址”状态，订单页的提示可直接进入补全，有地址订单不显示误报。

**Acceptance Scenarios**:

1. **Given** 订单录入候选缺少地址，**When** 显示确认卡，**Then** 该候选明确标记“待补地址”，同时仍允许桃子先保存草稿。
2. **Given** 订单页包含缺地址草稿，**When** 桃子查看列表，**Then** 该订单明确标记“待补地址”，且补全入口与订单位于同一操作上下文。
3. **Given** 订单已有有效地址，**When** 显示确认卡或订单页，**Then** 不显示缺地址状态。

### Edge Cases

- 地址为 `null`、空字符串或只含空白字符时，均视为缺地址。
- 顾客已有默认地址但旧草稿地址快照为空时，不在确认阶段静默回填；必须让桃子确认补全当前订单快照后再确认订单。
- 补地址保存与确认同时发生时，确认只依据当次操作读取到的订单地址快照；不得产生“订单仍缺地址但已确认”的中间状态。
- 补地址写入任一步失败时，顾客默认地址和当前订单地址快照都保持操作前状态。
- 已取消订单不因补地址而恢复；已经确认的历史缺地址数据不在本功能中自动补建送餐任务或改写状态。
- 地址补全不得修改其他顾客、其他订单或 `kith-inn-v1` 数据。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须允许缺地址订单以草稿状态保存和查询。
- **FR-002**: 系统必须把 `null`、空字符串和纯空白地址统一识别为缺地址。
- **FR-003**: 系统必须在所有订单确认入口执行同一缺地址守卫，不能依赖某个界面自行阻止。
- **FR-004**: 缺地址确认被拒绝时，系统必须保持订单为草稿，不打开相关餐次，不创建送餐任务，也不留下部分写入。
- **FR-005**: 缺地址确认被拒绝时，系统必须返回明确、可操作的“先补地址”提示，而不是通用失败信息。
- **FR-006**: 订单录入确认卡必须为缺地址候选展示“待补地址”状态，同时允许先保存草稿。
- **FR-007**: 订单页必须为缺地址草稿展示“待补地址”状态，并提供从该订单上下文直接开始补全的入口。
- **FR-008**: 地址补全必须拒绝空白值，并在一次成功操作中同时更新当前订单地址快照和对应顾客默认地址；任何通用更新入口不得绕过该原子操作修改订单地址。
- **FR-009**: 地址补全失败时，当前订单地址快照和顾客默认地址必须同时保持原值。
- **FR-010**: 更新顾客默认地址不得追溯改写该顾客的其他既有订单地址快照。
- **FR-011**: 顾客已有默认地址但目标订单快照仍为空时，确认不得静默借用默认地址；桃子必须先完成当前订单的显式补全。
- **FR-012**: 地址补全不得改变订单的生命周期状态；确认仍是补全后的独立动作。
- **FR-013**: 已有有效地址的订单必须继续遵循现有确认流程，不增加额外确认步骤。
- **FR-014**: 本功能只能影响 kith-inn 的顾客、订单、餐次和送餐数据，不得读取或修改 `kith-inn-v1` 业务数据。

### Key Entities

- **草稿订单**: 尚未进入经营口径的订单；包含顾客、日期、餐次和当前订单地址快照。地址快照是确认守卫的唯一地址事实。
- **顾客**: 保存以后新订单可带出的默认地址；默认地址更新不追溯改变既有订单快照。
- **餐次**: 订单确认后才可能打开的经营时间范围；缺地址拒绝时不得改变。
- **送餐任务**: 订单确认后才产生的履约记录；缺地址拒绝时不得创建，补地址后使用订单地址快照展示和排序。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 对空值、空字符串和纯空白地址的确认尝试，100% 被拒绝且订单、餐次、送餐任务零变化。
- **SC-002**: 点击确认和口头确认对缺地址订单的结果一致率为 100%，都给出可操作的补全提示。
- **SC-003**: 桃子从订单页的“待补地址”标识出发，可在一次进入补全、一次保存后回到可确认状态，无需离开当前订单上下文查找顾客。
- **SC-004**: 地址补全成功后，目标订单地址快照与顾客默认地址一致率为 100%，其他既有订单快照变化数为 0。
- **SC-005**: 含缺地址候选或草稿的订单录入确认卡与订单页，待补状态展示覆盖率均为 100%；有效地址订单误报率为 0。
- **SC-006**: 所有既有有地址订单的确认、送餐和收款行为保持通过，且本功能不产生任何 `kith-inn-v1` 文件或数据变化。

## Assumptions

- kith-inn 当前所有 confirmed order 都会进入送餐口径，因此确认前必须有订单地址快照；本功能不引入自取或无需地址的订单类型。
- 缺地址草稿本身是允许且有价值的中间状态，不要求在首次粘贴接龙或自然语言录入时强制补地址。
- “补地址”默认表示同时维护该顾客以后使用的默认地址和当前订单快照；若未来需要只改某单地址，应另开功能定义明确入口。
- 已确认的历史缺地址订单若存在，保留原状供人工处理；本功能不做数据迁移或自动修复。
- 依赖现有订单草稿、订单确认、顾客默认地址和送餐任务生命周期；不改变其他确认前置条件。
