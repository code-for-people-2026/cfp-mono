# User stories：街坊味（kith-inn）

> 目的：把 [`PRD.md`](./PRD.md) 的高层需求，和 [`prototype/kith-inn-high-fidelity-prototype.png`](./prototype/kith-inn-high-fidelity-prototype.png) / [`prototype/index.html`](./prototype/index.html) 的四个主入口，拆成可实现、可验收、可追踪的细化 user story。
>
> v1.10 说明：完整接龙按日期/餐次覆盖范围内全部 active 订单；自然语言补单只改唯一坐标，并区分追加量与最终总数。所有写入都先展示差异或运算，再确认。
>
> v1.9 说明：原型图仍用于说明四个 tab 的信息架构，但有三处被 PRD v1.9 覆盖：`今天`首屏不再固定展示概览；`送餐`不再按楼栋结构化/奶奶交接；菜品池 M1 只让用户维护菜名和主料，采购配方字段可以预留但不展示、不使用。

## 1. 角色与边界

### 1.1 桃子（MVP 唯一操作者）

- 她在微信里收接龙和私聊，下单信息不标准。
- 她做饭时手忙、偏语音，不适合多层菜单和小字表格。
- 她希望少漏单、少想菜单、少送错，而不是做复杂经营管理。

### 1.2 顾客（MVP 无界面）

- 顾客仍在微信群/私聊里下单和付款。
- MVP 不提供顾客端、不接支付、不要求顾客迁移。

### 1.3 明确不做

- 不做顾客侧点单、支付、订阅。
- 不做成本利润报表。
- 不做配送路线地图。
- 不通过软件协同奶奶职责。
- 不做忌口、喜好、二次加热等特殊要求字段。
- 不在 M1 UI 里维护菜品采购配方/用量；底层字段可预留给以后采购聚合。
- 不自建 ASR，MVP 走手机输入法语音键。

## 3. 全局壳与导航

### US-G01 进入桃子的灶台

作为桃子，我打开小程序后，要直接进入自己的“街坊味 / 桃子的灶台”，不需要选择商家或配置账号。

- 原型锚点：P0 topbar。
- PRD：§4.1 MVP 只有桃子一个使用者；§9 M0 手动 seed 桃子配置。
- 验收：
  - 进入后默认落在“今天”tab。
  - 只展示桃子自己的数据。
  - 不出现多商家选择、自助开通、复杂设置流程。

### US-G02 四个主入口稳定可达

作为桃子，我要始终能在底部切换“今天 / 菜单 / 订单 / 送餐”，这样我可以在对话和明细页之间来回核对。

- 原型锚点：P0 bottom tabbar。
- PRD：§5.5 一个主对话 + 三个确定性详情 tab。
- 验收：
  - 四个 tab 在所有主屏底部固定展示。
  - 菜单、订单、送餐 tab 放各自概览。
  - 今天 tab 不固定展示概览，只展示聊天。

### US-G03 底部输入框支持粘贴和语音

作为桃子，我要在底部输入框里粘接龙，也能点语音键用手机输入法说话，这样厨房里不用打很多字。

- 原型锚点：P0 composer。
- PRD：§3 语音优先；§6.1；§10 MVP 不自建 ASR。
- 验收：
  - 输入框接受长文本粘贴。
  - 语音按钮不启动自建录音识别，只调用平台输入能力或提示用输入法。
  - 输入内容进入“今天”主 agent 处理。

## 4. 今天：纯聊天 + 固定卡片

### US-T01 今天首屏保持干净聊天

作为桃子，我打开“今天”时，只看到聊天历史和输入框，不要一上来塞固定概览。

- 原型锚点：P1 聊天窗口；原型里的顶部固定指标废弃。
- PRD：§5.5 「今天」首屏规则。
- 验收：
  - 首屏不固定展示订单数、缺口数、菜单概览。
  - 聊天历史按时间顺序展示。
  - 概览入口在菜单/订单/送餐各自 tab 内。

### US-T02 查询类问题返回固定展示卡

作为桃子，我问“今天订单情况”“今天菜单是什么”“谁还没送”“谁没付款”时，要看到结构化卡片，而不是一段难核对的长文字。

- 原型锚点：P1 对话回复；P2/P3/P4 的卡片形态作为展示参考。
- PRD：§5.5 数据模型明确的展示类问题。
- 验收：
  - 订单查询卡展示日期、餐次、订单数、份数、未付数、草稿数。
  - 菜单查询卡展示日期、餐次、菜名、主料、发布状态。
  - 送餐查询卡展示待送订单、地址、份数、状态。
  - 卡片数据来自确定性查询，不依赖聊天记忆。

### US-T03 操作类问题返回确认卡

作为桃子，我说“明天晚餐王阿姨加 2 份”或“把明天午餐牛腩换掉”时，要先看到系统准备修改什么，再点确认。

- 原型锚点：P1 对话内动作卡。
- PRD：§5.5 数据操作类问题。
- 验收：
  - 确认卡必须列出资源类型、定位字段、动作、改前/改后。
  - 用户确认前不写库。
  - 确认后返回成功卡，并能跳到对应 tab。
  - 失败时说明失败资源，不吞掉部分错误。

### US-T04 信息不完整时回补全卡

作为桃子，如果我只说“加王阿姨 2 份”，系统不能猜日期和餐次，要问我补齐。

- 原型锚点：P1 对话内补全卡。
- PRD：§5.5、§6.1 修改/删除更严格。
- 验收：
  - 新增订单至少需要日期、餐次、顾客、数量。
  - 修改/删除至少需要日期、餐次、顾客、动作和目标值。
  - 定位到多条资源时让用户选择。
  - 不完整输入不得直接写入。

### US-T05 松散问题自然语言回答

作为桃子，我问没有固定资源卡片的问题时，系统可以自然语言回答；如果和经营无关，就礼貌挡回。

- 原型锚点：P1 普通气泡。
- PRD：§5.5 范围外话题。
- 验收：
  - 无固定资源的问题不强行造卡。
  - 范围外问题不回答泛知识，提示可帮忙记单、看菜单、查送餐。

### US-T06 展示历史只保留今天和昨天

作为桃子，我能看到今天和昨天的对话，更早的对话清理掉，但业务订单不能丢。

- 原型锚点：P1 顶部历史提示可保留。
- PRD：§5.5 三层记忆。
- 验收：
  - chat_messages 展示窗口按 Asia/Shanghai 保留今天+昨天。
  - 超 1000 条时删最旧 200 条。
  - orders/menu/fulfillments 不随聊天清理而删除。

## 5. 记单与订单写入

### US-OI01 接龙解析成候选变更卡

作为桃子，我把微信群接龙整段粘进来后，要看到系统准备新增、更新、退出或保留哪些订单，再确认写入。

- 原型锚点：P1 接龙识别卡。
- PRD：§6.1 输入与解析。
- 验收：
  - 每行候选包含日期、餐次、顾客、数量、地址状态。
  - 接龙标题的日期/周几决定范围；菜单正文不作为订单，也不与系统菜单比对。
  - 每行标记“新增 / 更新 / 退出 / 不变 / 待补”。
  - 用户确认前不写库。
  - 解析失败段落不瞎猜，进入待补。

### US-OI02 午餐晚餐可分别处理

作为桃子，如果一条接龙里写“王阿姨 1 份晚餐 2 份午餐”，系统要把午餐和晚餐分别识别、确认、付款、送餐。

- 原型锚点：P1 识别结果；P3 午餐/晚餐分区。
- PRD：§6.1 订单粒度。
- 验收：
  - 生成王阿姨 / date / lunch / quantity=2 的 order。
  - 生成王阿姨 / date / dinner / quantity=1 的 order。
  - 午餐和晚餐的状态、收款、送餐可独立变化。
  - 午餐和晚餐拆成两个 order，各自生成自己的 fulfillment。

### US-OI03 重复粘贴更新同坐标订单

作为桃子，我后来又粘贴同一天同一餐的完整接龙时，系统要以最后一次为准更新这一范围，而不是重复新增。

- 原型锚点：P1 候选变更卡；P3 订单台账。
- PRD：§6.1 重复粘贴 / 更新；§7 order 粒度。
- 验收：
  - 同一顾客+日期+餐次命中已有 order。
  - 覆盖范围包含此前由接龙或自然语言录入的全部 active order，不按来源分组。
  - 确认卡展示新增、改前/改后、退出和不变；确认前零写入。
  - confirmed order 被更新前必须二次确认。
  - paid/reconciled 或已送达的 order 不由完整接龙自动更新或取消，提示单独处理。
  - 取消状态的旧 order 不被静默复活，需明确提示。

### US-OI04 熟客自动带地址字符串

作为桃子，我希望熟客下单时自动带出默认地址字符串，这样接龙里没有地址也能进送餐清单。

- 原型锚点：P1/P3 地址展示。
- PRD：§6.1 地址；§6.4。
- 验收：
  - 已有顾客按名字归一匹配。
  - 匹配后展示 `defaultAddress` 字符串。
  - 地址作为 order.address 快照保存。

### US-OI05 新客补地址字符串

作为桃子，新客没有地址时，我要先补一个地址字符串，再确认该订单。

- 原型锚点：P1/P3 “缺地址/补地址”。
- PRD：§6.1 地址；§6.4。
- 验收：
  - 新客草稿可保存。
  - 缺地址的配送订单进入待补状态。
  - 补地址后写入 customer.defaultAddress，并回填 order.address。
  - 地址不拆楼栋/单元/房号字段。

### US-OI06 自家单不特殊建模

作为桃子，如果我把“桃子家”录进订单，它就和普通顾客一样处理，不需要特殊标识。

- 原型锚点：P3 订单行。
- PRD：§6.1 自家入单；§7 customers/orders。
- 验收：
  - 不写 `self` / `onsite` / 特殊顾客类型。
  - 桃子家只是一个 displayName。
  - 后续展示、收款、送餐都按普通订单规则处理，不靠自家特例排除。

### US-OI07 一句话补单必须信息完整

作为桃子，我可以说“7 月 3 日晚餐，王阿姨加 2 份”或“7 月 3 日晚餐，王阿姨改成 2 份”，系统先把理解的运算展示给我确认。

- 原型锚点：P1 操作确认卡。
- PRD：§6.1 私聊补单。
- 验收：
  - 输入包含日期、餐次、顾客、数量、动作。
  - “加 N 份”显示`当前数量 + N → 最终数量`；“改成/总共 N 份”显示`当前数量 → N`。
  - 原本没有记录时从 0 计算；原本已有记录时，“加”仍表示追加而不是覆盖。
  - 只修改该顾客+日期+餐次，不把同日其他订单列为退出。
  - 确认后写入 draft 或更新 existing order。

### US-OI08 一句话修改/删除必须唯一定位

作为桃子，我可以说“取消 7 月 3 日午餐李叔那单”或“7 月 3 日晚餐王阿姨改 1 份”，但系统必须先确认改的是哪条。

- 原型锚点：P1 操作确认卡。
- PRD：§5.5、§6.1 修改/删除更严格。
- 验收：
  - 修改数量展示 old quantity → new quantity。
  - 取消展示 order 状态将变为 canceled。
  - 找不到资源时不创建新单，提示未找到。
  - 多条可能命中时要求选择。

## 6. 订单 tab

### US-O01 查看订单概览

作为桃子，我进入“订单”tab 时，要看到当前日期的订单数、份数、未付数和草稿数。

- 原型锚点：P3 顶部概览。
- PRD：§5.5 概览放分 tab；§6.5。
- 验收：
  - 默认日期按 Asia/Shanghai。
  - 概览按餐次可切换/分组。
  - draft、confirmed、canceled 分开计算。

### US-O02 按餐次查看订单

作为桃子，我要按午餐/晚餐查看订单，每行包含顾客、份数、地址、收款状态。

- 原型锚点：P3 午餐/晚餐分区。
- PRD：§6.1 order = 顾客 + 日期 + 餐次。
- 验收：
  - 午餐和晚餐可分别筛选、确认和修改。
  - 同一顾客跨午晚时分别展示。
  - 每行可进入详情或触发快捷操作。

### US-O03 确认 draft order

作为桃子，我确认草稿没问题后，要能单条或批量确认，让它进入经营口径。

- 原型锚点：P3 draft 区。
- PRD：§6.1 确认 / 写入；§7 生命周期。
- 验收：
  - draft → confirmed 后打开对应 service_slot。
  - confirmed 后进入送餐/未付汇总。
  - 缺地址订单不能静默确认，必须提示。

### US-O04 手动切收款状态

作为桃子，我收到红包后，要能把对应订单标成已付；晚点核对后可标成已确认入账。

- 原型锚点：P3 “切收款”“标已付”。
- PRD：§6.5。
- 验收：
  - 支持 unpaid → paid → reconciled。
  - 支持误操作回退到 unpaid。
  - 收款状态不改变订单生命周期。
  - 未付汇总实时更新。

### US-O05 取消订单

作为桃子，我发现某人临时不要了，要能取消这单，并从送餐和未付缺口里消失。

- 原型锚点：P3 订单操作；P1 确认卡。
- PRD：§6.1；§7 canceled。
- 验收：
  - order.status 变为 canceled。
  - 对应 fulfillment 置 canceled。
  - 不计入份数、送餐缺口、未付汇总。
  - 历史仍可查，不物理删除。

## 7. 菜单 tab

### US-M01 查看菜单概览

作为桃子，我进入“菜单”tab 时，要看到本周或某天午/晚餐菜单。

- 原型锚点：P2 本周菜单。
- PRD：§6.2。
- 验收：
  - 默认展示当前周或最近餐次。
  - 每个菜单卡展示日期、餐次、菜名、主料、状态。
  - 概览不固定出现在今天首屏。

### US-M02 菜品池只维护菜名和主料

作为桃子，我维护菜品池时，只需要填菜名和主料，不需要填口味、费工、配方。

- 原型锚点：P2 菜品标签；v1.9 字段简化。
- PRD：§6.2 菜品池。
- 验收：
  - 新增菜只要求 name。
  - mainIngredient 可填可改。
  - M1 表单不出现口味、费工、忌口、recipe、ingredients 字段。
  - 底层可保留 recipe/ingredients 预留字段，但 M1 不读取、不要求填写。

### US-M03 自动排菜默认主料避重

作为桃子，我点自动排菜单时，希望系统默认尽量避免最近主料重复。

- 原型锚点：P2 “主料避重”概念保留。
- PRD：§6.2 确定性选菜内核。
- 验收：
  - 自动排菜只从菜品池选择。
  - 默认尽量避开近期主料和同菜重复。
  - 菜品池不足时提示，而不是发明新菜。

### US-M04 一键随机替换某道菜

作为桃子，我不想做某道菜时，要能一键随机换成另一道。

- 原型锚点：P2 “换一道”。
- PRD：§6.2 一键随机换菜。
- 验收：
  - 替换目标只影响选中的那道菜。
  - 默认仍尽量避开近期主料。
  - 替换结果来自菜品池。

### US-M05 用户指定换菜可覆盖避重

作为桃子，我明确说“把牛腩换成香菇滑鸡”时，即使主料重复，也应该允许我确认后替换。

- 原型锚点：P2 换菜；P1 菜单修改确认卡。
- PRD：§6.2 用户强制指定优先。
- 验收：
  - 系统提示可能破坏主料避重。
  - 用户确认后按指定菜替换。
  - 不因规则冲突阻拦用户。

### US-M06 在今天里口头改菜单

作为桃子，我可以在“今天”里说“明天午餐随机换掉牛腩”或“把明晚空心菜换成手撕包菜”，系统给我菜单修改确认卡。

- 原型锚点：P1 操作确认卡；P2 菜单卡。
- PRD：§6.2 今天可口头改菜单。
- 验收：
  - 确认卡定位到 date + occasion + menu item。
  - 随机替换和指定替换都支持。
  - 确认后更新 menu_plan。

### US-M07 生成发群文案

作为桃子，我确定菜单后，要一键生成能复制到微信群的文案。

- 原型锚点：P2 “发群文案”。
- PRD：§6.2。
- 验收：
  - 文案包含日期/餐次/菜名/价格或订餐提醒。
  - LLM 超时不影响菜单本身保存。
  - published 仅表示 App 内发布/可复制，不等于已发微信群。

## 8. 送餐 tab

### US-D01 查看按地址排序的待送清单

作为桃子，出餐后我要看到待送订单按地址字符串相似度排好，方便顺手送。

- 原型锚点：P4 待送清单；楼栋 chip 废弃。
- PRD：§6.3 地址相似度排序。
- 验收：
  - 列表展示顾客、份数、地址字符串、收款状态、送达状态。
  - 排序不依赖结构化楼栋字段。
  - confirmed 且未 canceled 的订单进入清单。

### US-D02 单条标送达

作为桃子，我送完某一单后，要能点这条订单标为送达。

- 原型锚点：P4 “送达”按钮。
- PRD：§6.3 灵活勾销。
- 验收：
  - 单条 fulfillment.status 变 done。
  - 其他订单不受影响。
  - 缺口数量实时减少。

### US-D03 地址片段标送达

作为桃子，我可以说“26B 送了”或输入地址片段，系统找出可能匹配的待送订单。

- 原型锚点：P0 composer；P1 确认卡。
- PRD：§6.3 地址片段。
- 验收：
  - 唯一命中时仍返回确认卡，展示将标记哪些订单。
  - 多命中时让用户选择或全选。
  - 不唯一时不得静默批量标 done。

### US-D04 收尾缺口

作为桃子，一趟送完后，我要看到还有哪些订单没标送达。

- 原型锚点：P4 收尾缺口。
- PRD：§6.3 收尾对账。
- 验收：
  - 缺口只统计 pending。
  - done/canceled 不算缺口。
  - 今天查询卡和送餐 tab 的缺口一致。

## 9. 顾客资料

### US-C01 记单时自动沉淀顾客

作为桃子，我不想单独维护顾客表；系统应在记单时自动沉淀顾客名字、默认地址、默认份数/餐次。

- 原型锚点：P1/P3 新客补地址、熟客自动带地址。
- PRD：§6.4。
- 验收：
  - 新顾客在补地址或确认订单时创建。
  - 老顾客按归一化名字匹配。
  - 不强制采电话。
  - 不做忌口/喜好/二次加热。

### US-C02 地址是一个字符串

作为桃子，我只想录“26B 703”这类地址文本，不想拆楼栋、单元、房号。

- 原型锚点：P1/P3 地址展示。
- PRD：§6.4。
- 验收：
  - customer.defaultAddress 是 string。
  - order.address 是 string 快照。
  - 无 building/unit 字段要求。

### US-C03 名字变体可人工处理

作为桃子，如果“王燕萍 / 王阿姨”其实是同一人，我要能在发现时修正，不要求系统自动猜所有别名。

- 原型锚点：P1/P3 订单行。
- PRD：§6.4。
- 验收：
  - 系统可做基础归一。
  - 低置信匹配不自动合并。
  - 可人工改订单关联顾客。

## 12. 通用验收规则

- 所有“今天”按 Asia/Shanghai 计算。
- 一个 order = 一个顾客 + 一个日期 + 一个餐次；午餐和晚餐是不同 order。
- draft 是纯记录：不开餐、不建 fulfillment、不进送餐、未付等经营口径。
- confirmed 才物化：打开 service_slot、生成 fulfillment、进入汇总。
- canceled 是终态：历史可查，但退出送餐/缺口/未付/份数口径。
- 同一天同餐同顾客再次写入是更新，不是新增重复数据。
- 地址是字符串；送餐排序按地址相似度，不依赖楼栋字段。
- 今天主对话和详情 tab 必须读同一套后端状态，不允许两套逻辑。
- 数据修改类对话必须先给确认卡；信息不完整不得写库。
- 菜单自动排菜默认主料避重，但用户强制指定时允许覆盖。
