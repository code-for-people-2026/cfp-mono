# User stories：街坊味 v1（kith-inn-v1）

> 状态：草稿 v0.1
>
> 日期：2026-07-09
>
> 输入：从旧 `kith-inn` 老板侧工具演进为“桃子分享可订餐小程序卡片 + 顾客自助下单 + 双端只读/可操作边界”的 v1 产品形态。

## 0. 可行性结论

### 0.1 顾客不显式登录下单：可行

产品上可以做到“顾客点小程序卡片后，不走登录/注册流程，直接进入订餐页”。技术上仍需要小程序调用 `wx.login` 拿临时 `code`，后端用 code 换取该用户在当前小程序内的 `openid`，作为顾客身份键。

注意边界：

- 能拿的是 **openid**，不是顾客的微信号、手机号、昵称。
- `openid` 是当前小程序维度的稳定身份；换小程序 appid、换微信号都不是同一个人。
- 不需要 `wx.getUserProfile`，也不要拿头像昵称；该接口每次都会弹授权，不符合“轻下单”。
- 顾客第一次提交时手填“称呼 + 送餐地址”；二者作为一条 `customer_profile` 绑定保存，不做称呼和地址的自由组合。后续同一 openid 再进来时，系统按 `openid + seller` 找到历史顾客资料，让他从 select 里选，或只有一条时默认选中。
- 称呼、地址、订单都是个人信息，提审时必须配置小程序用户隐私保护指引；表单提交前用简短文案告知用途。

### 0.2 分享小程序卡片：可行

桃子在商家侧选择若干餐次，系统生成一个“订餐批次”（如 `booking_batch`），然后通过小程序转发卡片分享出去。分享 path 带批次 id，例如：

```text
/pages/booking/index?batchId=abc123
```

顾客从卡片进入后，页面读取 `batchId`，拉取这批可订餐次，只能在这些餐次里选择份数。这里的 `batchId` 是对外随机 id，不使用数据库自增 id。

### 0.3 AI：MVP 可以一处都不用

本形态把旧版最需要 AI 的“接龙文本解析”砍掉了，改成顾客点 UI 下单。剩下核心能力都可以确定性实现：

- 批量录入菜品：用表格/多行文本的简单解析，不需要 AI。
- 随机生成菜单：只从菜品池抽取，用确定性规则避重，不需要 AI。
- 生成订餐卡片：表单选择餐次，不需要 AI。
- 顾客下单：表单，不需要 AI。
- 订单生命周期：按钮/状态机，不需要 AI。

因此 v1 MVP 直接砍掉 agent 对话功能。AI 只作为后置可选增强：从一大段历史菜单里解析菜名，或从旧微信群接龙补录历史订单；不进入主链路。

### 0.4 合规与审核风险

老板侧纯工具比顾客侧点单更容易过审。v1 一旦让顾客在小程序里订餐，审核上可能被视为餐饮/点餐相关场景；即使不接支付，也需要在小程序服务类目和食品资质上单独确认。MVP 仍不接支付，先把风险压到“预约/订餐信息收集 + 商家自配送记录”，但这是上线前必须复核的开放问题。

公开内容和按钮文案尽量统一为“预订登记”，避免使用“点餐”“外卖”“团购”“平台”“下单即付”等更容易触发餐饮/平台审核联想的词。产品实际能力仍要如实表达：顾客是在登记某几个餐次的预订意向，线下支付和履约由桃子自己处理。

兜底方案：如果顾客侧小程序卡片下单无法通过审核，退回“老板侧工具 + 微信群接龙”形态。桃子仍在微信群发接龙，系统提供接龙文案导入：从接龙文本解析日期、餐次、用户名、份数生成订单；接龙里没有地址，不要求补地址，不进送餐地址资料，由桃子按线下记忆处理。

### 0.5 本文依据

- 微信开放文档 [`wx.login`](https://developers.weixin.qq.com/miniprogram/dev/api/open-api/login/wx.login.html)：可获取登录凭证 code，并由后端换取 openid/session_key。
- 微信开放文档 [`code2Session`](https://developers.weixin.qq.com/miniprogram/dev/OpenApiDoc/user-login/code2Session.html)：小程序登录凭证校验接口路径为 `/sns/jscode2session`。
- 微信开放文档 [`Page.onShareAppMessage`](https://developers.weixin.qq.com/miniprogram/dev/reference/api/Page.html#onShareAppMessage-Object-object)：页面可自定义转发标题、path、图片。
- 微信开放文档[“转发”](https://developers.weixin.qq.com/miniprogram/dev/framework/open-ability/share.html)：页面内 `button open-type="share"` 可触发分享。
- 微信开放文档 [`wx.getUserProfile`](https://developers.weixin.qq.com/miniprogram/dev/api/open-api/user-info/wx.getUserProfile.html)：获取用户信息需要点击触发，且每次请求都会弹授权。
- 微信开放文档[“用户隐私保护指引”](https://developers.weixin.qq.com/miniprogram/dev/framework/user-privacy/)：涉及处理用户个人信息的小程序需补充隐私保护指引。

## 1. 角色与边界

### 1.1 桃子（商家）

- 她维护菜品池，生成每日/每周菜单。
- 她选择可订餐次，分享订餐小程序卡片到微信群。
- 她查看所有订单，处理订单生命周期：确认、标已付、标已送、取消。

### 1.2 顾客

- 顾客从微信群里的小程序卡片进入。
- 顾客不注册、不填手机号、不授权头像昵称。
- 首次使用填称呼和送餐地址；后续从历史资料中选择。
- 顾客只能看自己的订单和状态，不能改订单生命周期。

### 1.3 明确不做

- 不解析微信群接龙作为主链路。
- 不做 agent 聊天入口。
- 不接支付，不做在线退款。
- 不做顾客头像、昵称、手机号采集。
- 不做配送路线/地图。
- 不做成本利润、采购用量、忌口偏好。
- 不做多商家自助开通；MVP 仍只服务桃子。

## 3. 全局与身份

### US-G01 进入商家侧

作为桃子，我打开小程序后默认进入商家侧，这样我可以直接管理菜单和订单。

验收：

- 桃子通过已绑定的 operator 身份进入商家侧。
- 商家侧只展示桃子的菜品、订餐批次、订单。
- 顾客从分享卡片进入时不看到商家后台入口。

### US-G02 顾客静默识别

作为顾客，我点开微信群里的小程序卡片后，不想注册登录，只想直接订餐。

验收：

- 顾客页面启动后调用 `wx.login`，后端换取 openid。
- openid 只作为身份键，不展示给顾客。
- 获取 openid 失败时不允许提交订单，提示“微信身份识别失败，请重进小程序”。
- 不调用 `wx.getUserProfile`，不弹头像昵称授权。

### US-G03 顾客资料按 openid 复用

作为顾客，我第一次填过称呼和地址后，之后不想每次重复填写。

验收：

- 首次提交前必须填写称呼和送餐地址。
- 系统保存 `openid + seller + displayName + address` 的顾客资料；`displayName + address` 是一条 profile，不拆成两个可自由组合的字段。
- 再次进入时展示历史资料 select，选项文案形如“王阿姨 · 3A-1201”。
- 只有一条历史资料时默认选中；多条时让顾客选择。
- 顾客可以新增一个资料，例如“妈妈家 / 26B-301”。
- 同一顾客多个地址保存为多条 profile；同一地址多人下单也保存为多条 profile，避免误组合。
- 顾客侧只展示绑定当前 openid 的资料。
- 桃子替尚未进入小程序的私聊顾客创建资料时，openid 可以为空；这类资料只在商家侧可见，不能按相同称呼或地址自动认领给某个微信用户。

## 4. 菜品池与菜单

### US-M01 批量录入菜品

作为桃子，我想一次粘贴多道菜，这样不用一道一道新增。

验收：

- 菜品录入支持多行文本，每行至少一个菜名。
- 可选填写主料和分类，例如“番茄牛腩 牛肉 荤”。
- 系统在保存前展示解析预览。
- 重名菜提示跳过或覆盖，不静默生成重复菜。
- 不使用 AI；只做简单分隔符解析。

### US-M02 手动维护菜品

作为桃子，我想随时新增、编辑、停用菜品，这样菜单池保持可用。

验收：

- 菜品字段只包含菜名、主料、分类、启用状态。
- 停用菜不参与后续菜单生成。
- 已用于历史菜单的菜品停用后，历史菜单仍能显示菜名快照。

### US-M03 生成某一餐菜单

作为桃子，我想给某个餐次随机生成菜单，这样不用每天想吃什么。

验收：

- 桃子选择日期 + 餐次后点击生成。
- 菜单只从启用菜品池选择。
- 默认结构为 2 荤 2 素 1 汤，可先写死为桃子的配置。
- 尽量避开最近做过的同菜和同主料。
- 菜品池不足时明确提示，不发明新菜。

### US-M04 批量生成一周菜单

作为桃子，我想一键生成一周午餐/晚餐菜单，这样前一晚不用临时想。

验收：

- 桃子选择一段日期和餐次范围。
- 系统按餐次批量生成菜单。
- 同一周内尽量不重复同一道菜；同一天尽量不重复同主料。
- 生成后可逐餐调整。
- 已发布订餐的餐次重新生成前必须二次确认。

### US-M05 换一道菜

作为桃子，我对某餐里一道菜不满意时，可以点一下换掉。

验收：

- 只替换目标菜，不重排整餐。
- 替代菜来自同类菜品池。
- 默认避开当前餐已出现的主料。
- 没有可替代菜时提示补菜品池。

## 5. 分享订餐

### US-S01 创建订餐批次

作为桃子，我想选择几个可订餐次，比如今天中午、今天晚上、明天晚上，然后生成一个可分享的订餐入口。

验收：

- 桃子可以从日期/餐次列表里多选。
- 每个餐次展示菜单、价格、截止时间和状态。
- 只能选择未关闭且允许订餐的餐次。
- 保存后生成 `booking_batch`。

### US-S02 分享小程序卡片

作为桃子，我想把订餐批次分享成小程序卡片发到微信群，这样顾客点开就能订。

验收：

- 分享按钮调用小程序转发能力。
- 分享 path 包含 `batchId`；`batchId` 是随机公开 id，不暴露数据库 id。
- 卡片标题用日期 + 星期 + 餐次表达清楚，例如“7月9日 周四 午餐/晚餐预订”。
- 顾客打开卡片后只能看到该批次内的餐次。

### US-S03 关闭订餐批次或餐次

作为桃子，我想在截止后关闭某个餐次，避免继续来单。

验收：

- 订餐批次是落库记录 `booking_batches`，保存本次分享选择的多个 `meal_slots`、分享标题、状态和创建时间。
- “关闭整个批次”指将该 `booking_batch.status` 置为 `closed`，这张分享卡片入口整体停止接受新增订单。
- 桃子也可以只关闭批次中的某一个餐次，即关闭对应 `meal_slot` 的订餐入口。
- 关闭后顾客仍可看自己的历史订单，但不能新增该餐次订单。
- 已有订单不被自动取消。

## 6. 顾客下单

### US-C01 打开分享卡片

作为顾客，我点开群里的卡片后，要看到这次可以订哪些餐。

验收：

- 页面展示商家名、可订餐次、菜单、价格。
- 已截止/已关闭餐次置灰。
- 如果 `batchId` 无效，展示“这个订餐链接已失效”。

### US-C02 选择资料

作为顾客，我想从上次用过的称呼和地址里选择，这样不用重复填。

验收：

- 有历史资料时，顶部展示资料 select。
- 选择资料后自动带出称呼和地址。
- 顾客可以编辑本次地址；编辑后可选择是否保存为新资料。
- 没有历史资料时直接展示新资料表单。
- 不提供“先选称呼、再选地址”的自由组合 UI；每个选项都是一条完整资料。

### US-C03 选择餐次和份数

作为顾客，我想在一个页面里选自己要订哪些餐次、每餐几份。

验收：

- 每个可订餐次有份数 stepper，默认 0。
- 份数为 0 表示不订该餐。
- 至少一个餐次份数大于 0 才能提交。
- 提交前展示确认摘要：称呼、地址、餐次、份数、总价。

### US-C04 提交订单

作为顾客，我确认后提交订单，这样桃子能在后台看到。

验收：

- 提交按 `seller + mealSlot + customerProfile` 幂等写入；顾客请求还必须校验 profile 绑定当前 openid。
- 同一顾客资料对同一餐次再次提交时，更新份数而不是重复新增。
- 份数改为 0 时需要明确确认，确认后取消该餐次订单。
- 提交成功后进入“我的订单”结果页。

### US-C05 查看自己的订单

作为顾客，我想看到我刚订的订单状态，这样知道桃子有没有确认、我有没有送达。

验收：

- 顾客只能看到当前 openid 下自己的订单。
- 展示订单状态、付款状态、送达状态。
- 顾客不能点击标已付、标已送、确认订单等商家操作。
- 顾客提交后订单先是 `draft`，表示桃子还没确认。
- 桃子点“确认订单”后订单变为 `confirmed`，表示锁单。
- 商家确认前且未过截止时间，顾客可以自助改份数或取消。
- 锁单后顾客不能自助修改或取消，只能主动联系桃子。
- 商家已确认后，顾客不能自助修改，提示“桃子已确认，请在群里联系桃子”。
- 餐次已关闭或已过截止时间时，顾客不能自助修改，提示“本餐次已截止，请在群里联系桃子”。

### US-C06 顾客改份数或取消

作为顾客，如果我刚才填错份数或临时不要了，我想在桃子确认前自己改掉。

验收：

- 顾客只能修改自己 openid 下的订单。
- 仅允许修改 `status=draft` 且对应餐次未截止、未关闭的订单。
- 顾客可以把份数改为新的正整数。
- 顾客可以取消订单；取消前二次确认。
- 改份数或取消后，商家侧订单列表实时反映。
- 已 confirmed、canceled、已截止或已关闭餐次的订单不可由顾客自助修改。

## 7. 商家订单生命周期

### US-O01 查看订单总览

作为桃子，我想看到每个餐次的订单数、总份数、未付、未送，这样知道今天要做什么。

验收：

- 商家侧按日期 + 餐次展示订单。
- 每个餐次展示总订单数、总份数、未付数、未送数。
- 支持从订餐批次进入，也支持按日期查看。
- 顾客新提交的订单默认是 `draft`，需桃子确认后才进入锁单/备餐口径。

### US-O02 查看单个餐次订单

作为桃子，我想看某餐所有订单，这样方便备餐和送餐。

验收：

- 订单行展示顾客称呼、份数、地址、付款状态、送达状态。
- 默认按地址字符串排序。
- 支持按地址/称呼搜索过滤。
- canceled 订单灰显，不计入总份数和缺口。

### US-O03 确认订单

作为桃子，我想确认顾客提交的订单，这样确认后顾客不能再自助改动，我可以放心备餐。

验收：

- 桃子可以单条确认订单。
- 确认后 `order.status` 从 `draft` 变为 `confirmed`。
- `confirmed` 订单进入备餐总份数、未付、未送等经营口径。
- 顾客侧同步显示“桃子已确认”。
- 确认后顾客不能自助改份数或取消，只能联系桃子。
- 桃子仍可在商家侧修改或取消 confirmed 订单，修改前提示会影响备餐/送餐清单。

### US-O04 手动添加订单

作为桃子，我想在商家侧手动添加订单，这样自己家吃饭或顾客私聊下单时，不用再走顾客分享卡片。

验收：

- 桃子可以选择日期、餐次、份数，填写或选择顾客资料。
- 顾客资料仍按“称呼 + 地址”一条 profile 保存；可选已有资料，也可新建资料。
- 私聊顾客尚未进入小程序时，新 profile 可以没有 openid；不得根据称呼或地址猜测绑定微信身份。
- 自己家下单不特殊建模，例如可建一条 profile：“桃子家 · 自家不送”。
- 手动添加的订单和顾客自助提交的订单进入同一订单列表、同一生命周期。
- 若命中同一 `customerProfile + mealSlot` 的订单，提示“更新现有订单”而不是静默新增重复订单。
- 手动添加订单的来源标记为 `manual`，方便以后排查。
- 手动订单只有在 profile 已绑定 openid 且订单保存相同 `customerOpenid` 时才出现在该顾客的“我的订单”；无 openid 的手动订单只在商家侧可见。

### US-O05 手动修改订单

作为桃子，我想修改手动补的单或顾客自助提交的单，这样私聊改份数、改地址时能及时同步。

验收：

- 桃子可以修改份数、称呼/地址快照、备注。
- 改称呼/地址快照默认只影响当前订单。
- 如需保存为常用资料，提供“保存为新资料”或“更新此资料”，更新此资料需二次确认。
- 修改 confirmed 订单前提示会影响备餐/送餐清单。

### US-O06 标已付

作为桃子，我收到顾客红包后，可以把订单标为已付。

验收：

- unpaid → paid。
- paid 可回退 unpaid。
- 付款状态变化不影响送达状态。
- 顾客侧只读同步看到付款状态。

### US-O07 标已送达

作为桃子，我送完后可以勾选订单标为已送达。

验收：

- 支持单条标已送达。
- 支持逐行勾选后批量标已送达。
- done 可回退 pending。
- 顾客侧只读同步看到送达状态。

### US-O08 取消订单

作为桃子，我发现某单不做了，可以取消。

验收：

- order.status 变为 canceled。
- canceled 订单退出未付、未送、总份数口径。
- 顾客侧能看到“已取消”。
- 历史保留，不物理删除。

### US-O09 导出或复制备餐清单

作为桃子，我想复制某餐的备餐/送餐清单，这样厨房和配送时能核对。

验收：

- 清单包含餐次、总份数、顾客、份数、地址。
- 可复制纯文本。
- 不生成路线规划。

## 8. 顾客只读生命周期

### US-RO01 订单状态说明

作为顾客，我想看懂订单处于什么状态。

验收：

- 展示三条轴：订单状态、付款状态、送达状态。
- 文案使用顾客能懂的词：已提交、桃子已确认、已付款、已送达、已取消。
- 不展示内部 id、openid、状态枚举值等技术字段。

### US-RO02 隐私与资料管理

作为顾客，我想知道我填的称呼和地址用来做什么。

验收：

- 首次填资料时展示短文案：“用于桃子识别订单和送餐地址”。
- 提供“删除本机/当前微信下保存资料”的入口。
- 删除资料是软停用，不物理删除；历史订单不受影响，后续下单不再带出。

## 9. 数据规则

- `seller`：商家/租户，MVP 只有桃子。
- `operators`：v1 商家侧身份，绑定 v1 seller 和微信 openid；不作为共享 Payload Admin 用户。
- `customer_profiles`：顾客在某商家下的一条“称呼 + 地址”资料；openid 可空，只有绑定当前 openid 的资料才在顾客侧出现。
- `offerings`：菜品池。
- `meal_slots`：某个 `YYYY-MM-DD` 日历日的午餐/晚餐，同时保存菜单快照、价格、截止时间和预订状态。
- `booking_batches`：一次分享直接关联的多个 meal slots。
- `orders`：一个顾客资料 + 一个 meal slot 的套餐份数，同时保存确认、付款、送达三条状态轴和称呼/地址快照。
- MVP 不建立独立 `menu_plans`、`booking_batch_slots`、`order_items`、`fulfillments`；出现多商品或多履约任务的真实需求后再拆表。

核心约束：

- 同一 `seller + mealSlot + customerProfile` 只保留一条 order；重复提交或取消后重订都复用该记录。
- 顾客重复提交同餐次是更新，不是新增。
- 桃子手动添加同坐标订单也是更新确认，不是新增重复订单。
- 顾客只能读 `customerOpenid` 等于当前 openid 的订单；不按称呼或地址匹配订单归属。
- 桃子能读写自己 seller 下全部订单。
- 顾客无生命周期写权限。
- 顾客只能在订单 `draft` 且餐次未截止/未关闭时改份数或取消；不能标已付、标已送、确认订单。
- `confirmed` 表示商家已确认并锁单；顾客修改/取消必须主动联系桃子。
- `customer_profile` 绑定称呼和地址；MVP 不做称呼表、地址表自由组合。
- `orders` 必须保存 `displayName` 和 `address` 快照，后续 profile 修改不影响历史订单。

## 10. AI 使用决策

### US-AI01 主链路不使用 AI

作为产品维护者，我希望 v1 主链路不依赖 AI，这样系统更稳定、更可测，也少一个失败点。

验收：

- 没有 agent 聊天入口。
- 没有“粘接龙解析成订单”的主流程。
- 菜单生成用确定性算法。
- 顾客下单用表单。
- 订单生命周期用按钮和状态机。

### US-AI02 AI 只作为后置导入工具

作为桃子，如果我以后要导入旧接龙或历史菜单，可以接受 AI 帮忙预解析，但不能影响日常下单主链路。

验收：

- AI 导入工具默认不出现在 MVP 主导航。
- AI 结果必须预览确认后才写库。
- AI 失败不影响分享订餐、订单查看和菜单生成。

### US-AI03 审核兜底：接龙文案导入订单

作为产品维护者，如果顾客侧下单因为小程序类目/资质审核无法上线，我需要保留一个老板侧兜底入口，让桃子把微信群接龙粘进来生成订单。

验收：

- 兜底入口默认不作为 MVP 主链路，仅在顾客侧下单无法上线时启用。
- 输入是一段微信群接龙文案，文案里应包含日期、餐次、顾客名、份数。
- 系统解析出订单草稿，并给桃子预览确认后写入。
- 接龙文案里没有地址时，不阻塞写入；订单地址可为空。
- 地址为空的订单不自动创建或更新 `customer_profile`。
- 无地址订单在送餐清单里标记“无地址”，由桃子线下记忆处理。
- 解析不出日期或餐次时，不写库，提示桃子补充。

## 11. 非 MVP Backlog

| Story | 阶段 | 说明 |
|---|---|---|
| 顾客在线支付 | 后置 | 涉及类目/资质/支付，不进 MVP |
| 固定订阅 | 后置 | 周/月固定餐次自动生成订单 |

## 12. 通用验收规则

- 所有“今天/明天”按 Asia/Shanghai 计算。
- 顾客侧不显式登录，不拿手机号、昵称、头像。
- openid 只能作为后端身份键，不暴露给用户。
- 顾客首填称呼和地址，之后从历史资料选择；一条历史资料 = 一个称呼 + 一个地址。
- 桃子可创建无 openid 的私聊顾客资料；此类资料和订单默认只在商家侧可见。
- 订餐批次只允许提交所选餐次。
- 已关闭餐次不能新增订单，但历史订单可查看。
- 顾客只读自己的订单生命周期；唯一写权限是截止前修改/取消自己的 draft 订单。
- 商家确认订单后锁单，顾客侧不可再自助修改或取消。
- 桃子商家侧可操作订单状态。
- AI 不进入 MVP 主链路。
- 顾客侧下单若审核不可行，兜底切回接龙文案导入订单。
- 顾客可见文案优先使用“预订登记”，避免“点餐/外卖/团购/平台”等公开表述。
